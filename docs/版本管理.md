
# 版本管理方案

## 概述

本项目采用**统一版本号**方案管理前后端代码的版本。所有包（根目录、desktop、server）使用相同的版本号，确保前后端版本一致。

## 方案选择

### 为什么选择统一版本号？

对于前后端紧密耦合的 Electron 桌面应用，统一版本号具有以下优势：

- **简单直观**：一个版本号代表整个产品
- **避免混乱**：不会出现前后端版本不一致的问题
- **易于理解**：团队成员和用户都能轻松理解版本关系
- **便于发布**：发布时只需更新一个版本号

### 其他可选方案

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **独立版本号** | 前后端可独立迭代发布 | 需要管理多个版本号，可能出现兼容性问题 | 前后端团队完全分离、独立部署 |
| **混合版本号** | 兼具灵活性和统一性 | 复杂度较高 | 大型项目，部分包独立发布 |

## 项目结构

```
rag-demo/
├── package.json          # 根目录（版本号唯一来源）
├── apps/
│   ├── desktop/
│   │   └── package.json  # 桌面端（版本号与根目录同步）
│   └── server/
│       └── package.json  # 服务端（版本号与根目录同步）
└── scripts/
    └── sync-version.js   # 版本同步脚本
```

## 使用方法

### 版本更新命令

在根目录执行以下命令：

| 命令 | 说明 | 示例 |
|------|------|------|
| `pnpm version:patch` | 更新补丁版本（bug修复） | 0.1.0 → 0.1.1 |
| `pnpm version:minor` | 更新次版本（新功能） | 0.1.0 → 0.2.0 |
| `pnpm version:major` | 更新主版本（重大变更） | 0.1.0 → 1.0.0 |
| `pnpm version:sync` | 手动同步版本号到所有子包 | - |

### 完整发布流程

1. 更新版本号
   ```bash
   pnpm version:patch
   ```

2. 提交更改
   ```bash
   git add .
   git commit -m "chore: bump version to x.y.z"
   ```

3. （可选）创建标签
   ```bash
   git tag vx.y.z
   git push origin vx.y.z
   ```

## 版本号规范

遵循 [语义化版本 2.0.0](https://semver.org/lang/zh-CN/)：

- **主版本号**：不兼容的 API 修改
- **次版本号**：向下兼容的功能性新增
- **补丁版本号**：向下兼容的问题修正

## 文件说明

### scripts/sync-version.js

版本同步脚本，负责将根目录的版本号同步到所有子包。

**功能：**
1. 读取根目录 `package.json` 的版本号
2. 更新 `apps/desktop/package.json` 的版本号
3. 更新 `apps/server/package.json` 的版本号

## 注意事项

1. **始终在根目录执行版本命令** - 不要直接修改子包的版本号
2. **提交前先测试** - 确保更新版本号后项目能正常构建和运行
3. **Git 标签** - 建议为每个重要版本创建 git tag，便于追溯
