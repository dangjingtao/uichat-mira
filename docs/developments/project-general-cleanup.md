## 目标

在“项目-通用”设置页提供一个“一键清理”入口，让用户集中删除当前用户的全部对话、应用工作空间记录、聊天附件和媒体产物，并清空服务端日志。

本功能只描述用户可见的产品行为和清理边界。各类数据的真实存储位置、后端接口和桌面端调用方式必须以实现阶段核对到的源码为准。

## 清理范围

点击“一键清理”后，处理当前登录用户的全部对话及其消息记录，删除应用工作空间记录，清理应用管理的媒体目录，并清空服务端日志：

| 数据类型 | 说明 |
| --- | --- |
| 历史对话 | 当前用户的活动或已归档对话 |
| 对话消息 | 被删除历史对话关联的消息记录 |
| 服务端日志 | `server.log` 和 `error.log` 的现有内容 |
| 应用工作空间 | 工作空间记录、名称和路径，不包含根目录中的文件 |
| 聊天附件 | 应用附件目录中的聊天附件 |
| 图片产物 | 图片生成应用管理的本地图片产物 |
| 音频产物 | TTS 输出和参考音频 |
| 视频产物 | 应用管理的视频产物目录（当前不存在时不产生删除项） |

当前系统没有独立的项目 ID，本功能按当前登录用户隔离数据。不得删除其他用户的对话。

## 保留范围

以下内容不属于本功能的默认清理范围：

- 活动中的对话
- 工作空间根目录中的普通文件
- 项目配置和项目元数据
- 用户导入的原始资料
- 知识库原始文件
- 日志、缓存、临时文件和任务产物
- 其他用户的数据

“历史对话”指聊天侧边栏中展示的对话记录，包括活动和已归档状态。工作空间本身必须保持可用。

## 交互流程

1. 用户在“项目-通用”中点击“一键清理”。
2. 系统展示二次确认弹窗。
3. 弹窗明确说明：本次操作会删除当前用户的全部对话及其消息，并清空服务端日志，不会修改工作空间。
4. 用户取消时关闭弹窗，不执行任何清理。
5. 用户确认后开始执行清理。
6. 清理完成后展示结果。

确认按钮必须是明确的破坏性操作按钮，不能通过点击入口后直接执行删除。

## 结果反馈

执行结束后，页面至少展示：

- 清理是否完成
- 删除的对话数量
- 删除的消息数量
- 未删除成功的数量
- 失败原因（当前实现能够提供时）

如果部分数据清理失败，已经成功处理的内容不应被伪装成整体失败；结果需要区分成功项和失败项，并允许用户看到失败范围。

如果没有可清理内容，也应展示明确结果，例如“当前项目没有可清理内容”，不能让用户只能通过按钮状态猜测执行结果。

## 行为边界

- 清理操作只处理对话记录，不删除工作空间；清理期间不应启动新的清理请求。
- 清理操作不得删除工作空间或修改工作空间配置。
- 清理操作不得删除工作空间根目录中的普通文件、缓存或其他未列入清理范围的任务产物。
- 清理操作只处理当前登录用户的数据。
- 单项删除失败时，结果必须可见。
- 清理过程中应避免重复提交；执行中的入口需要表现为不可重复触发。
- 清理操作永久删除对话，不提供恢复入口。

## 验收标准

1. “项目-通用”页面存在“一键清理”入口。
2. 点击入口不会直接删除数据，必须先经过二次确认。
3. 确认提示明确说明删除历史对话及其消息，并明确工作空间不会被修改。
4. 取消确认不会改变任何数据。
5. 清理只针对当前登录用户的对话。
6. 活动对话和工作空间记录不会被清理或修改。
7. 文件、日志、缓存和任务产物不会被清理。
8. 清理完成后能看到成功、无可清理内容或部分失败等明确结果。
9. 部分失败时，界面能区分已清理内容与未清理内容。
10. 清理执行期间不能重复发起同一操作。
11. 实现和测试不得依赖硬编码本地路径；测试临时数据库和产物放在仓库根目录 `.test-artifact/` 下。

## 当前未决事项

- 清理失败后是否提供单项重试。
