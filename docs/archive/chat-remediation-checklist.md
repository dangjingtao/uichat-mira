> process and is kept for implementation history only. Current runtime rules
> now live in `docs/uchat.md`, `docs/uchat-internal-maintenance.md`, and

本文件用于跟踪本轮 chat 主链路整改的执行原则、范围与完成状态。

使用规则：

- 每开始一项改动前，优先检查本地代码是否已经实现该能力。
- 如果代码已经实现且行为正确，不重复改造。
- 如果代码未实现，或已实现但行为不符合当前产品规则，再进入修改。
- 每完成一项整改，并且通过对应测试后，再把该项从 `- [ ]` 改为 `- [x]`。
- 前端工程验收不由 Codex 代替，统一交由项目 owner 手动测试。
- 当代码整改完成且协议/单测通过后，Codex 停下来，明确列出需要项目 owner 手测的功能清单，再进入前端人工验收。

## 当前执行原则

- `uchat` 是唯一聊天主实现，不新增新的 legacy chat 主路径。
- 欢迎态与已持久化线程必须彻底分离。
- `knowledgeBaseId` 是 chat 是否走 RAG 的唯一真相源。
- 修改前先查现有代码、类型、本地文档和测试，不按猜测改。
- 优先做最小必要修复；如果现有实现已满足目标，不为“重写得更漂亮”而改。
- 先保证状态一致性、持久化一致性和协议一致性，再处理表现层细节。
- 前端工程验收由项目 owner 手测；Codex 只负责代码修改、协议测试、单元测试和明确列出手测项。
- 线程标题优先由助手首轮回复完成后使用 `task` 模型生成；若标题模型不可用或生成失败，则回退为用户第一句话。

## 暂记 UI Bug

以下问题已记录，作为后续实现参考；当前回合先不单独修：

- 发送按钮在发送中应切为“可点击、可取消”的状态，而不是单纯禁用
- `runStatus` 目前只有 `idle / running / error`，没有可中断的取消契约；“发送中可取消”属于架构级改动，需先由项目 owner 确认后再改
- 助手处于发送中状态时，消息显示仍不正常；当前已部分改善，输出结束后显示会恢复正常，但流式过程中的展示仍需继续修
- 欢迎态切回已有线程后，旧线程内容基本不被污染，但消息输入框未同步更新
- RAG 流程完成后，线程标题没有及时更新到当前 UI
- 欢迎态首发发送创建了新线程，但新线程 UI 没有及时正确显示第一条消息
- 右侧参考来源/知识库抽屉打开后的输入区布局修复存在副作用：当前正常状态下输入区发生偏移，说明抽屉展开态与常态的宽度协调还需重做
- 图片发送入口当前未真正限制为图片文件；需继续核实现有上传前是否已接入压缩，以及压缩发生在哪一层
- 线程视图滚动跟随策略仍不正确：当前切换线程时会自动滚到底部；预期应只在用户原本停留在底部且流式更新时才保持贴底。切线程时不应出现平滑动画
- 消息复制当前已可用，但缺少前端成功/失败反馈
- `uchat` 核心能力仍未实现：重新生成回复、删除消息、编辑消息；该组能力待当前 bug 收口后继续接入
- 输入框缺少默认快捷键机制：应支持 `Ctrl+Enter` 直接发送消息
- 输入框缺少贴图能力：`Ctrl+V` 粘贴图片时，应直接把图片加入待发送附件区
- 输入框发送按钮状态语义不明确：需清晰区分“可发送 / 禁用 / 发送中可取消”等状态
