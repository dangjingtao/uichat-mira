# Knowledge Base MVP

Layer: raw-source
Module: KnowledgeBase
Feature: MvpHistory
Doc Type: historical

## 目标

在当前项目中，用最小可行范围实现一套可用的单知识库 RAG 流程：

1. 上传文档
2. 解析文本
3. 切分 chunk
4. 向量入库
5. 检索命中
6. 在聊天中展示命中片段

MVP 的重点是先打通闭环，而不是一次性做成完整知识库平台。

---

## 本次确认后的 MVP 范围

### 保留

- 单知识库
- 上传文档
- 文档列表页
- 文档详情页
- 文档检索
- 聊天接入检索结果
- 引用展示

### 当前收敛

- 文件类型只支持 `md` / `txt`
- 引用展示优先支持：
  - `文档名 + 命中片段`
- 如果实现复杂度过高，可先退一步只展示：
  - `命中片段`

### 暂不做

- 多知识库
- Excel / PDF / DOCX 解析
- 文档级权限
- 真正的异步任务队列
- 批量复杂操作
- 多种高级分段策略
- 重排、重写、多路召回等高级检索能力

---

## 为什么先不做这些

### 文档级权限

文档级权限指的是：

- 不同用户只能查看或检索自己有权限的文档

这会引入额外设计：

- 文档和用户 / 角色绑定
- 列表页权限过滤
- 检索时权限过滤

对当前 MVP 来说，这一层会明显增加复杂度，因此先不做。

### 真正的异步任务队列

真正的异步任务队列指的是：

- 上传接口只负责创建任务
- 解析、切分、向量化在后台 worker 中完成
- 前端通过轮询或推送拿任务状态

这会引入：

- 任务表
- 状态流转
- 失败重试
- 后台 worker
- 前端轮询 / 推送

当前只做 `md/txt`，文件处理成本相对较低，因此第一版可以先走同步处理或轻量伪异步处理。

### rerank

当前项目已经支持“可选 rerank”。

它的作用仍然是：

1. 先用向量检索召回一批 chunk
2. 再用 rerank 模型把这些 chunk 重新排序

当前实现的约束是：

- 只有在默认 rerank 模型已配置且启用时，主流程才会调用 rerank 服务
- 当前适配器走 OpenAI-compatible `/v1/rerank` 接口，适合 LocalAI 一类兼容服务
- 未配置、未启用或调用失败时，会自动回退到原始向量检索排序

它仍然适合这些场景：

- 文档量较大
- chunk 很多
- 纯向量召回质量不稳定

如果没有合适的 rerank 服务，当前阶段仍然建议先做：

- 合理切分
- 合理 topK
- 稳定的向量检索

## 建议的 MVP 页面

### 1. 知识库列表页

用途：

- 展示所有文档
- 查看状态
- 进入详情页
- 进入上传向导

建议字段：

- 名称
- 分段模式
- 字符数
- 召回次数
- 上传时间
- 状态

### 2. 添加文件向导

建议保留 3 步，但第一版功能可以很轻：

#### Step 1：选择数据源

- 选择 / 拖拽文件
- 当前只支持 `md` / `txt`

#### Step 2：文本分段与清洗

第一版可以只给一个固定策略，或者少量可选项：

- chunk size
- overlap
- 是否去掉空行

#### Step 3：处理并完成

展示：

- 上传成功 / 失败
- 文档名
- chunk 数
- 处理耗时

### 3. 文档详情页

展示：

- 文档基本信息
- 标签
- 上传时间 / 更新时间
- 分段预览
- 处理状态

## 建议的后端最小能力

### 上传链路

1. 接收 `md/txt` 文件
2. 解析文本
3. 执行固定切分
4. 生成 embedding
5. 写入 sqlite / sqlite-vec
6. 返回处理结果

### 检索链路

1. 输入 query
2. embedding query
3. 向量检索 topK
4. 返回命中文档片段

### 聊天链路

1. 用户提问
2. 先检索知识库
3. 将命中片段拼入 prompt
4. 调用模型生成回答
5. 返回答案 + 引用片段

---

## 建议的数据结构

第一版建议至少有两张核心表。

### documents

字段建议：

- `id`
- `name`
- `source_type`
- `file_type`
- `status`
- `char_count`
- `chunk_count`
- `created_at`
- `updated_at`

### document_chunks

字段建议：

- `id`
- `document_id`
- `chunk_index`
- `content`
- `embedding`
- `created_at`

如果当前项目里向量存储已经有既定结构，可以按现有结构适配，不强行重构。

---

## 引用展示建议

第一版建议接口直接返回完整引用结构：

```ts
type Citation = {
  documentId: string;
  documentName: string;
  snippet: string;
  score?: number;
};
```

这样前端可以：

- 第一版只显示 `snippet`
- 第二版很容易升级为 `文档名 + 命中片段`

建议不要把接口先做窄，否则后面还要补字段。

---

## 不带 rerank 的检索建议

### 建议先做的参数

- 固定 chunk size
- 固定 overlap
- 固定 topK

例如：

- chunk size：`400 ~ 800` 字符
- overlap：`50 ~ 120` 字符
- topK：`3 ~ 5`

最终取值要结合中文文档内容长度和模型上下文来调。

### 比 rerank 更值得优先优化的点

- chunk 切分方式
- overlap 大小
- topK
- prompt 中如何组织检索结果
- 返回给前端的引用信息是否清晰

---

## 实施顺序建议

### 阶段 1：数据与接口

- 明确 `documents` / `document_chunks`
- 明确上传接口
- 明确检索接口

### 阶段 2：上传闭环

- 上传 `md/txt`
- 解析、切分、入库
- 列表页可看到新文档

### 阶段 3：检索闭环

- 根据 query 返回命中 chunk

### 阶段 4：聊天接入

- 聊天请求中接入检索
- 返回答案与引用

### 阶段 5：详情与优化

- 文档详情页
- 处理结果页
- 更好的引用展示

---

## 当前推荐结论

当前项目的知识库 MVP，建议严格控制在下面这个范围：

- 单知识库
- 仅支持 `md/txt`
- 同步上传处理
- 向量检索
- 聊天显示命中片段
- 最好带上文档名

先把这条主链路跑通，再考虑：

- 更多文件类型
- 更复杂的分段策略
- rerank
- 权限
- 真异步任务
