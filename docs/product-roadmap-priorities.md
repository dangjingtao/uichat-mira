# 产品方向优先级与实现难度评估

这份文档用于沉淀当前 4 个产品方向的判断，给后续版本排期、拆阶段和讨论取舍提供一个统一参考。

适用范围：

- 桌面端本地 RAG 调试主产品
- 当前已有 React + Electron + Fastify + SQLite 技术栈
- 近期目标仍以“把核心主链路做扎实”为主，而不是同时铺开多个重型系统

## 评估结论

当前建议的优先级排序：

1. 聊天界面功能完善
2. 评测工作台增强
3. 工具 / Skill / 沙箱执行体系
4. 外接服务能力

当前建议的实现难度排序：

1. 聊天界面功能完善：中
2. 评测工作台增强：中偏高
3. 外接服务能力：中偏高到高
4. 工具 / Skill / 沙箱执行体系：高

这里的“优先级”不是在说谁最酷，而是在说：

- 谁更直接提升当前产品可用性
- 谁更贴近现有代码与能力边界
- 谁更容易形成完整闭环并被持续验证

## 四个方向逐项判断

### 1. 聊天界面功能完善

典型范围包括：

- 发送图片
- 重新生成回答
- 用户编辑 / 撤回对话
- 更完整的消息态与分支态管理

优先级：最高

难度：中

原因：

- 这是用户最直接感知的主界面，收益立刻可见。
- 这条线和现有产品定位最贴近，不需要先引入新的系统边界。
- 现有项目已经基于 assistant-ui 构建聊天体验，很多能力有成熟参考路径，不是从零起步。
- 这类功能补齐后，会直接抬高产品“完成度感”，也能为后续工具调用、评测回看、多模态输入打基础。

主要工作内容：

- 补齐附件上传与消息渲染链路
- 梳理消息编辑、重新生成、分支切换的数据结构
- 统一消息操作区交互
- 处理输入态、失败态、加载态、取消态等边角体验

主要风险：

- 看起来像小功能，实际容易碰到线程状态、消息版本、流式响应恢复等细节。
- 如果前端状态结构没收好，后续再接工具调用会比较痛。

建议节奏：

- 先做“用户最常点到”的功能闭环：附件、重试 / 重生成、编辑消息
- 再补消息分支、撤回策略、失败恢复等偏结构性能力

### 2. 评测工作台增强

典型范围包括：

- 评测页面补更多诊断细节
- 样本级、attempt 级结果更好展示
- 引入评测模型打分
- 区分启发式分数与 judge 分数

优先级：第二

难度：中偏高

原因：

- 评测是这个产品和普通聊天壳子真正拉开差异的一条线。
- 当前项目已经有工作台、历史记录、报告导出、基础指标，这意味着它不是概念验证，而是可以继续往深处做。
- 相比工具系统，这条线更贴近现有数据模型和后端架构，扩展成本更可控。

主要工作内容：

- 丰富 run 详情与样本诊断视图
- 增加筛选、对比、排序、失败定位能力
- 设计 judge 模型调用协议、结果持久化与复算策略
- 明确“快速工程分”和“正式评测分”的产品表达

主要风险：

- 一旦引入 judge 模型，成本、时延、可重复性都会成为真实问题。
- 评分标准如果没有层次设计，用户会把启发式分和 judge 分混为一谈。
- 结果解释性不足时，评测页会显得“有数字但不可行动”。

建议节奏：

- 第一阶段先补展示与诊断能力，让已有结果更有用
- 第二阶段再接 judge 模型，并把结果来源、置信度、耗时和成本说清楚

### 3. 工具 / Skill / 沙箱执行体系

典型范围包括：

- 接入命令行工具
- 提供沙箱隔离环境
- 用户上传工具
- 定义内置 skill
- 形成 tool-skill 工作流，让 AI 真正替用户执行任务

优先级：第三

难度：最高

原因：

- 这是想象空间最大的一条线，但系统复杂度也最高。
- 它不是单点功能，而是一整套运行时能力：权限、隔离、工具协议、生命周期、失败恢复、日志、资源限制、结果展示，全都要一起考虑。
- 一旦做浅了，会变成“能演示，但不敢用”；一旦做深了，就是一个接近 agent runtime / local automation platform 的独立产品方向。

主要工作内容：

- 工具协议与执行模型
- 本地命令执行与权限边界
- 用户自定义工具注册 / 上传 / 描述 / 调试
- Skill 编排与提示词注入
- 任务日志、工件、回放、错误处理

主要风险：

- 安全边界和可控性是核心难点，不是锦上添花。
- UI、后端、运行时三边都要一起进化，任何一边薄弱都会拖垮体验。
- 容易与聊天主链路、评测链路、外部集成链路形成耦合。

建议节奏：

- 不要一上来做“大而全”平台
- 更适合先选 1 到 2 个高价值内置能力做成闭环，例如：
  - 代码生成并预览网页
  - 调用本地脚本处理文档
- 等工具协议、工件展示、失败恢复跑顺后，再开放用户上传工具

### 4. 外接服务能力

典型范围包括：

- 数据库接入
- 飞书、微信等三方服务
- 其它业务系统连接能力

优先级：第四

难度：中偏高到高

原因：

- 这条线价值很高，但更依赖明确场景。
- 如果没有优先行业、优先对象和典型工作流，外接服务很容易演变成“接了很多，但每个都不深”。
- 它通常要处理认证、权限、字段映射、失败重试、审计与连接状态，这些问题都偏琐碎但很重。

主要工作内容：

- 统一连接器模型
- 认证与凭据管理
- 服务能力清单与调用协议
- 与聊天、工具、知识库、评测之间的联动方式

主要风险：

- 每接一个服务都会带来长期维护成本。
- 没有统一抽象时，集成数量越多，系统越容易碎。
- 很多外接能力其实应该建立在工具体系之上，否则会出现重复建设。

建议节奏：

- 把外接服务当成“在明确场景中落能力”，不要当成单独的大盘子先铺开
- 优先选 1 个最有代表性的服务验证模式，再决定是否抽象成连接器层

## 为什么不是先做工具体系

工具体系很诱人，因为它看起来最能体现“AI 真能干活”。但从当前阶段看，它不应该排第一，主要有 3 个原因：

1. 当前产品的主体验和差异化还没有完全收稳
2. 工具体系一旦开工，工程面会迅速膨胀
3. 如果聊天主链路、评测链路还不够扎实，工具体系接上去以后也缺少稳定承载面

更实际的做法是：

- 先把聊天体验补齐到“每天都愿意用”
- 再把评测做成“出了问题知道怎么看、怎么比、怎么改”
- 然后用少量高价值内置工具试探 agent 化边界

## 推荐推进顺序

### 阶段一：先补核心使用体验

目标：

- 让聊天界面从“能用”走到“完整”
- 让评测结果从“有数据”走到“可行动”

建议包含：

- 聊天附件、重新生成、编辑消息
- 评测详情增强、筛选、对比前的基础整理

### 阶段二：把评测做深

目标：

- 建立更可信的质量反馈闭环

建议包含：

- judge 模型打分
- 分数来源分层
- 历史 run 对比与问题定位

### 阶段三：试点工具执行能力

目标：

- 用少量闭环场景验证 tool-skill 的真实价值

建议包含：

- 1 到 2 个内置工具场景
- 工件展示
- 执行日志
- 失败恢复

### 阶段四：按场景接入外部服务

目标：

- 让系统能进入真实业务流，而不是只停留在本地调试

建议包含：

- 从一个高价值连接器开始
- 沉淀统一连接模型，再决定扩展范围

## 如果只选一个近期重点

如果近期只能押一个方向，我会建议：

先做“聊天界面功能完善 + 评测工作台增强”这两个方向的组合，而不是单押其中一个。

原因很简单：

- 聊天界面负责入口体验
- 评测工作台负责结果闭环

两者组合起来，才能更像一个完整产品，而不是一个聊天壳子，或者一个只有工程师才会进的调试页。

## 结论

从当前阶段看，最合理的节奏不是“追最有想象力的方向”，而是先把主路径做厚：

1. 聊天界面补齐关键能力
2. 评测工作台做深做透
3. 用试点方式引入工具 / Skill / 沙箱体系
4. 再按场景推进外部服务集成

这样做的好处是，产品每前进一步都会更完整，也更容易验证下一步是不是值得继续加码。
