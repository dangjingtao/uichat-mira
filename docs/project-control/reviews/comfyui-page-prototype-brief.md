# ComfyUI 页面原型说明

Status: Draft  
Owner: product / design / frontend  
Last updated: 2026-07-08

## 这份文档干什么

这份文档只服务一件事：

把“微应用-生图”页面，收敛成一份可交给设计师继续出稿的 ComfyUI 页面原型说明。

它不讨论：

- 后端接口怎么设计
- provider 体系怎么重构
- 数据库存什么表
- 其它生图服务商怎么统一

今天只聚焦 ComfyUI。

---

## 页面定位

这不是一个“通用生图后台配置台”。

这页首先应该是一个：

**让用户围绕 ComfyUI 的 flow 完成配置、选择、运行和查看结果的工作台。**

用户来到这页，最核心的任务不是看很多卡片，也不是理解内部状态。

用户的真实任务应该是：

1. 先确认 ComfyUI 连得上
2. 选择一个可用 flow
3. 填写这个 flow 允许填写的输入项
4. 运行
5. 看结果
6. 必要时再看诊断信息

也就是说，这页的主角不是“provider 抽象”，而是：

- ComfyUI 连接
- Flow
- Flow 输入
- 执行结果

---

## 页面气质

建议整体气质是：

- 冷静
- 干净
- 工具化
- 有掌控感
- 不炫技

不要做成：

- AI 炫光风
- 通用后台卡片拼盘
- 一屏很多状态块同时吵用户
- 看起来像开发调试台

用户进来后，应该很快知道自己当前在做哪一步，而不是被一堆“摘要 / 状态 / 日志 / 帮助”同时拉扯注意力。

---

## 设计目标

这页要解决 4 个问题：

1. 用户能不能一眼看懂“先做什么”
2. 用户能不能明确知道“当前使用的是哪个 ComfyUI、哪个 flow”
3. 用户能不能专注填写 flow 输入，而不是被次级信息打断
4. 用户生成失败时，能不能自然进入排查，而不是一开始就被日志包围

---

## 页面结构总原则

页面应该从“全量铺开”改成“主流程优先，次级信息后置”。

建议采用下面这个结构：

1. 顶部：页面标题和一句用途说明
2. 主体左侧：配置与执行区
3. 主体右侧：结果预览区
4. 底部或折叠区：诊断与详情

阅读顺序必须自然形成：

`连接 -> flow -> 输入 -> 运行 -> 结果 -> 排查`

---

## 页面区块说明

## 1. 顶部区

### 目标

只负责告诉用户：

- 这是 ComfyUI 工作台
- 这里可以配置连接、管理 flow、运行生成

### 建议内容

- 页面标题：`ComfyUI 工作台`
- 一句说明：例如“连接本地或远程 ComfyUI，选择 flow，填写输入并运行生成”

### 不建议保留的东西

- 重复的大横幅
- 重复出现的标题和副标题
- 多个 badge 同时强调“实验中 / prompt + workflow”
- 大面积渐变英雄区

顶部应该轻，不要抢走主任务区的注意力。

---

## 2. 左侧主工作区

左侧是整页最重要的区域。

它应该承担“从准备到提交”的完整链路。

建议拆成 3 个连续区块。

### 2.1 连接区

#### 目标

让用户明确自己当前连的是哪个 ComfyUI，以及这个连接当前是否可用。

#### 应包含的信息

- 当前连接名称
- 地址
- 当前状态
  - 未配置
  - 已配置但未校验
  - 可连接
  - 连接失败
- 一个明确动作
  - 选择连接
  - 新建连接
  - 编辑连接
  - 测试连接

#### 设计要求

- 这里要像“当前工作环境确认”
- 不是大表单先铺一堆字段
- 用户应该先知道“我现在连的是谁”

#### 视觉建议

- 单块清晰的信息卡
- 状态明确
- 右上角或行尾有轻量操作

---

### 2.2 Flow 区

#### 目标

让用户选择、查看和切换 flow。

这是 ComfyUI 页面真正的核心。

#### 应包含的信息

- 当前选中的 flow 名称
- flow 简介或备注
- 最近更新时间
- flow 来源
  - 本地上传
  - 手动录入
  - 模板复制
- 一个主要动作
  - 选择 flow
- 次级动作
  - 上传新 flow
  - 新建 flow
  - 编辑 flow
  - 查看原始 JSON

#### 设计要求

- 这里不是只放一个大文本框让用户粘 JSON
- 用户应该有“我当前正在使用哪个 flow”的稳定感
- flow 是资产，不是一次性临时输入

#### 视觉建议

- 可以做成“当前 flow 卡 + 切换入口”
- 或做成“选择器 + 当前 flow 摘要”
- 不建议默认把完整 JSON 大面积暴露在首屏

---

### 2.3 Flow 输入区

#### 目标

让用户填写这次执行真正可改的内容。

#### 应包含的信息

- 这次执行的可编辑输入项
- 每个输入项的名称、用途、占位说明
- 默认值
- 必填 / 选填

#### 优先展示的输入类型

- 正向提示词
- 负向提示词
- seed
- 尺寸
- 参考图
- 模型相关可选项
- flow 自定义字段

#### 设计要求

- 这里只放“真的会生效”的字段
- 不要为了好看塞很多未来可能支持但当前无效的字段
- 输入项顺序要贴合真实使用频率

#### 视觉建议

- 让用户感觉像在填写“本次任务参数”
- 优先保证可读和节奏，不要为了紧凑把表单压得太碎

---

### 2.4 提交动作区

#### 目标

让用户明确知道“现在可以运行”、“正在运行”还是“需要先补条件”。

#### 应包含的信息

- 主按钮：`开始生成`
- 次级按钮：`重置本次输入`
- 运行状态提示
  - 可提交
  - 输入不完整
  - 正在生成
  - 生成失败

#### 设计要求

- 主动作必须非常明确
- 不要把“帮助、日志、请求摘要”跟主按钮放在同一视觉层级

#### 关于取消

如果当前版本真实不支持取消，就不要把“取消”做成正式主动作。

可以没有，也可以作为明确的“暂不支持”弱信息处理，但不要假装它是一个可用按钮。

---

## 3. 右侧结果区

右侧应该成为“本次生成结果”的专属区域。

它不应该再被拆成多个同权重的小卡片互相抢位置。

### 目标

让用户一眼知道：

- 结果出来没有
- 出来了什么
- 当前这次生成是成功还是失败

### 建议内容

- 大预览图
- 结果状态
- 关键元信息
  - 生成时间
  - 使用的 flow
  - 任务编号
  - 输出尺寸

### 成功态

成功时，预览图应该绝对是视觉中心。

其它信息都只做辅助，不要把图片压成一个次级小模块。

### 失败态

失败时，右侧不要只显示“失败”。

要显示：

- 失败说明
- 简短原因
- 一个进入诊断区的入口

### 空态

空态不要写得太技术。

重点是告诉用户：

- 还没有开始生成
- 先去左侧选择 flow 并填写输入

---

## 4. 诊断与详情区

这部分必须降级，不应占据首屏主战场。

建议放在页面底部，或者做成折叠区。

### 应放在这里的内容

- 请求摘要
- 任务状态时间线
- 调试日志
- 原始 JSON
- 失败诊断信息

### 设计原则

- 默认收起或弱化
- 只有在用户需要复盘、排障、核对细节时再展开

这部分是“支撑用户排查”，不是“默认打断用户生成”。

---

## 页面优先级

整页视觉优先级建议如下：

### 一级优先

- 当前连接
- 当前 flow
- 可编辑输入
- 开始生成
- 结果预览

### 二级优先

- 当前状态
- 最近一次执行结果
- flow 简介
- 关键元信息

### 三级优先

- 请求摘要
- 调试日志
- 帮助说明
- 原始 JSON

如果三级信息和一级信息在视觉上差不多重，说明页面结构还是错的。

---

## 页面交互节奏

建议设计师按下面这个节奏组织页面：

### 场景 1：首次进入

用户先看到：

- 这是 ComfyUI 工作台
- 还没有配置连接，或尚未选择 flow

页面主动作应该引导：

- 先配置连接
- 再选择或上传 flow

### 场景 2：已有连接，未选 flow

用户先看到：

- 连接正常
- 需要先选一个 flow

页面主动作应该引导：

- 去选 flow

### 场景 3：已选 flow，准备运行

用户先看到：

- 当前连接
- 当前 flow
- 一组明确的输入项
- 一个明显的“开始生成”

### 场景 4：运行中

用户先看到：

- 当前正在生成
- 结果区处于等待状态

运行中页面不应再让很多无关信息跳出来抢焦点。

### 场景 5：生成完成

用户先看到：

- 结果图
- 本次结果概览

如果失败，再给出进入诊断区的入口。

---

## 设计禁忌

以下方向建议明确避免：

1. 不要把页面做成很多等权重卡片的拼盘
2. 不要让日志、帮助、摘要在首屏和主输入区抢注意力
3. 不要把 JSON 文本框继续当作 flow 管理方案本身
4. 不要在顶部再做一块很重的装饰横幅
5. 不要把“不可用动作”做成像可用动作一样醒目的按钮
6. 不要把内部技术概念直接当页面主文案

---

## 给设计师的关键词

如果需要一句话传达给设计师，可以这样说：

> 这不是通用 AI 配置页，而是一个围绕 ComfyUI flow 的生成工作台。请用“连接确认、flow 选择、输入填写、结果预览、诊断后置”的结构来设计，主流程要比调试信息更醒目。

---

## 最终交付期待

希望设计稿最终能回答清楚这几个问题：

1. 用户进入页面后，第一步做什么
2. 用户如何知道当前连的是哪个 ComfyUI
3. 用户如何知道当前跑的是哪个 flow
4. 用户如何专注填写这次执行需要的输入
5. 用户如何快速看到结果
6. 用户失败时如何进入排查，而不是一开始就被调试信息包围

如果这 6 个问题在设计稿里都能一眼回答出来，这页就基本走对了。
