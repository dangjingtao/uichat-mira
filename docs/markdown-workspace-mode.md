# Markdown 工作空间能力评估

面向当前产品形态，对“允许用户定义工作空间文件夹，并在 Markdown 文档集合上执行轻量检索、读取、分类识别、重建整理、验证”的能力做产品判断、边界定义与 MVP 收敛。

## 结论

这个能力值得做，但前提是把它定义为一个**轻量文档工作模式**，而不是第二套 RAG 系统。

当前项目已经有完整的 RAG 主链路。新增 Markdown 工作空间能力的价值，不在于再次提供“知识库问答”，而在于提供一种**不需要入库、不需要分块、不需要 embedding，就能直接处理一组 Markdown 文档**的工作方式。

这类能力适合作为当前产品的补充模式，而不适合作为一个独立的大系统继续膨胀。

## 为什么值得做

### 1. 与现有 RAG 形态不冲突

现有 RAG 解决的是知识库问答、检索链路调试、评测与运行观测。

Markdown 工作空间解决的是另一类更轻的任务：

- 用户指定一个文件夹
- 系统只关注其中的 Markdown 文档
- 直接做文档检索、阅读、归类、整理和校验

它不是“再做一个知识库”，而是“让用户直接拿原始文档集合做知识工作”。

### 2. 显著降低使用门槛

很多用户并不想先完成下面这些准备动作：

- 导入知识库
- 分块
- 生成 embedding
- 配置检索链路
- 调整召回参数

他们的真实需求只是：

- 给一组 Markdown 文档
- 搜内容
- 读上下文
- 归类整理
- 检查整理结果是否靠谱

对于这类需求，现有 RAG 路径太重，而 Markdown 工作空间模式足够轻。

### 3. 符合桌面工作台定位

当前产品本身就是桌面端知识与 RAG 调试工作台。

在这个前提下，支持“本地文件夹 -> Markdown 集合 -> 研究整理”是自然延伸：

- 与桌面环境契合
- 与文件系统接近
- 与用户已有文档工作流接近

它能把产品从“RAG 测试器”轻推向“知识工作台”。

## 为什么不能把它做成第二套 RAG

这个能力一旦失控，价值会迅速下降。

以下方向不建议做：

- 为工作空间单独建设复杂的分块、embedding、重排、索引治理体系
- 为工作空间单独复制一遍完整知识库生命周期
- 把所有复杂认知动作都做成底层工具，导致工具边界失真
- 把它做成“文件夹版聊天”，但没有真实的整理与验证能力

如果这样演化，产品会出现两套相互竞争的知识系统：

1. 正式 RAG 知识库
2. 临时 Markdown 工作空间

这会让用户理解成本、工程复杂度和维护负担一起上升。

## 建议的产品定义

建议把这项能力正式定义为：

**Markdown 工作空间模式**

特征如下：

- 用户手动指定一个文件夹作为工作空间
- 系统只处理其中的 Markdown 文档
- 不要求入库
- 不要求预分块
- 不要求 embedding
- 面向轻量知识处理，不替代现有 RAG

## 核心实现原则

这项能力的底座，不应该先定义为一组过早抽象的业务工具，而应该先定义为：

**让 AI 在用户指定的工作空间中，受控地直面命令行。**

原因如下：

- Markdown 文档集合本质上仍然是文件系统中的原始文件
- 检索、读取、批量处理、结构提取、验证，天然都可以通过命令行原语组合完成
- 命令行比过早封装的高层业务工具更轻、更通用、更接近真实工作流

因此，这项能力的第一层应该是**受控命令行工作底座**，第二层才是面向 Markdown 的轻业务能力。

## 需要实现的基础工具

以下工具是这套模式要成立时必须具备的基础工具。这里的“基础”指系统原语和底层能力，不是业务层的“搜索 / 分类 / 整理”按钮名称。

### 1. `command_exec`

作用：

- 在受限工作空间中执行系统命令

这是整个能力的底座。没有它，系统只能依赖预先写死的高层工具，灵活性会大幅下降。

它支撑的上层场景包括：

- 枚举文件
- 搜索文本
- 读取局部内容
- 调用脚本做批处理
- 运行外部 CLI 或内部处理程序

### 2. `process_io`

作用：

- 获取命令执行后的 `stdout`
- 获取命令执行后的 `stderr`
- 获取命令退出码 `exit code`

没有这层，命令执行就不可观察，也无法判断成功、失败和失败原因。

### 3. `working_directory_scope`

作用：

- 明确定义命令执行的工作目录
- 保证所有动作默认限制在用户选定的工作空间内

这是“工作空间模式”的语义核心，不是一个可选优化项。

### 4. `path_guard`

作用：

- 校验路径是否逃逸工作空间
- 阻止绝对路径、`..` 路径或越权访问

它是文件访问和命令执行的共同安全边界。

### 5. `file_read_raw`

作用：

- 稳定读取原始文件内容

虽然很多读取动作可以通过命令行完成，但保留一个明确的原始读文件能力，能让系统在关键路径上更稳定，也更容易被上层复用。

### 6. `file_write_raw`

作用：

- 把整理结果、临时产物或输出文件写回工作空间

如果未来这项能力要支持“重建整理结果落盘”，这个能力就是必要原语。

### 7. `text_search_raw`

作用：

- 在文件集合中执行最基础的文本匹配

它可以由命令行工具承载，例如 `rg`、`grep` 等，但在体系设计上应被视为一个独立的基础能力，因为：

- 检索是高频动作
- 需要稳定的输入输出约定
- 后续很可能需要统一封装结果格式

### 8. `structured_parse`

作用：

- 把原始 Markdown 文本解析为结构化表示

至少应能识别：

- 标题层级
- 段落
- 列表
- 代码块
- 表格
- 引用块

没有这层，后续的分类识别、重建整理和验证都会退化成对纯文本的大段处理，效果和稳定性都会下降。

## 边界能力

为了让“让 AI 直面命令行”这件事可控，这套模式必须同时具备下面这些边界能力。

### 1. 工作空间边界

- 任何命令执行都必须绑定到明确的工作空间根目录
- 不允许默认落到进程当前目录或系统任意目录

### 2. 路径边界

- 禁止跨出工作空间的路径访问
- 禁止未授权路径写入
- 禁止通过软链、相对路径逃逸等方式绕过限制

### 3. 命令边界

- 明确允许执行哪些命令
- 明确禁止执行哪些命令
- 第一阶段更适合采用允许列表，而不是完全开放

### 4. 权限边界

- 区分只读能力与可写能力
- 默认应优先只读
- 只有在确实需要输出结果时才开启写入

### 5. 资源边界

- 命令执行必须有超时
- 输出必须有长度上限
- 必要时限制可处理的文件数量或文件大小

### 6. 可观测性边界

- 每次命令调用都应可追踪
- 需要能回看执行命令、工作目录、退出码和关键输出

这是后续做验证和排障的基础。

## 上层业务能力如何建立

在上面的底座具备之后，面向用户的轻业务能力才有意义：

- 检索
- 读取
- 分类识别
- 重建整理
- 验证

这些能力不应该直接作为第一层基础设施存在，而应该建立在上面的系统原语和边界能力之上。

更准确地说：

- “检索”来自 `text_search_raw`
- “读取”来自 `file_read_raw` 和受控命令行
- “分类识别”来自模型对结构化内容的处理
- “重建整理”来自模型和脚本对多文档材料的综合
- “验证”来自命令输出、结构化检查和模型复核的组合

## 不建议第一期做的能力

为了控制范围，以下内容不建议纳入 Markdown 工作空间第一期：

- 自动向量化
- 自动 chunk 管理
- 混合召回
- rerank
- 工作空间级评测体系
- 复杂后台任务编排
- 与正式知识库完全打通的数据同步

这些都更适合留给现有 RAG 主链路，而不是塞进这个轻模式。

## 建议的最小工作流

当前产品最适合的 Markdown 工作空间工作流如下：

1. 用户选择工作空间文件夹
2. 系统建立工作空间边界与命令执行范围
3. 系统扫描其中的 Markdown 文档
4. 用户或智能体执行文本检索
5. 系统读取目标文档或标题区块
6. 模型执行分类识别或重建整理
7. 系统结合命令结果与结构化内容做基本验证
8. 输出整理结论或待补充问题

这个流程的重点不是“快速回答”，而是“帮助用户对一组 Markdown 文档完成一次可追踪的知识处理”。

## MVP 建议

如果只做最小可用范围，建议收敛为：

### 必做

- 工作空间文件夹选择
- 受控命令执行能力
- 命令输出与退出码采集
- 工作空间与路径边界控制
- Markdown 文档扫描
- 文本检索
- 文档全文/区块读取
- Markdown 结构解析
- 基于文档集合的分类识别
- 基于文档集合的整理输出
- 基于任务目标的结果验证

### 可延后

- 标签缓存
- 目录树视图增强
- 整理结果导出
- 工作区历史记录
- 与主聊天线程联动

## MVP 计划表

下面这张表按“先打底座，再接文档能力，最后串成可用工作流”的顺序安排。目标不是一步到位做成复杂系统，而是尽快跑通第一条可用主链路。

| 阶段 | 核心目标 | 需要完成的内容 | 交付物 | 验收标准 |
| --- | --- | --- | --- | --- |
| P0 | 建立受控工作空间底座 | 工作空间根目录选择；工作目录作用域；路径越界校验；受控命令执行；采集 `stdout` / `stderr` / `exit code`；超时与输出长度限制 | 可在指定工作空间内稳定执行只读命令的底层能力 | AI 或后端服务能在指定目录下执行命令；无法越界访问；失败时能返回清晰错误 |
| P1 | 跑通 Markdown 文档发现与读取 | 扫描 Markdown 文件；读取文件元信息；读取全文；支持按标题或行范围切片；基础 Markdown 结构解析 | 工作空间内 Markdown 文档清单与文档读取能力 | 能列出工作空间内 Markdown 文档；能读取整篇文档；能读取 section；结构解析结果可用 |
| P2 | 跑通轻量检索主链路 | 基于文件名、路径、标题、正文关键词的检索；统一命中结果格式；从命中结果回读文档上下文 | Markdown 文档检索能力 | 对给定关键词能返回候选文档和命中位置；用户可从结果继续读取上下文 |
| P3 | 跑通第一次智能整理 | 基于检索与读取结果生成分类识别、摘要、提纲或专题整理；先支持 1 到 2 个固定整理模式 | 第一版“分类识别 / 重建整理”能力 | 对一组 Markdown 文档能输出结构化整理结果，而不是只返回原文拼接 |
| P4 | 跑通验证闭环 | 对整理结果做覆盖性、依据充分性、遗漏和冲突检查；输出“通过 / 不通过 / 待补充” | 第一版验证能力 | 系统能指出当前整理结果是否足以交付，并说明缺口 |
| P5 | 串成用户可感知的最小工作流 | 把工作空间选择、检索、读取、整理、验证串成一条完整流程；补最小 UI/交互反馈 | Markdown 工作空间 MVP | 用户可以从选择文件夹开始，完成一次“检索 -> 阅读 -> 整理 -> 验证”的闭环任务 |

## 推荐开发顺序

建议严格按下面的顺序推进，不要跳步：

1. 先做 P0
2. 再做 P1
3. 然后做 P2
4. P3 和 P4 可以并行收敛，但仍应建立在 P2 之上
5. 最后做 P5，把前面能力收束成可感知产品流程

这个顺序的关键点是：

- 没有 P0，后面所有能力都不稳
- 没有 P1，工作空间里没有可用文档对象
- 没有 P2，整理与验证都缺少稳定输入
- 没有 P3 和 P4，这个模式就只是一套文件浏览与搜索

## 第一阶段明确不做的事

为了保证 MVP 能收敛，第一阶段建议明确不做：

- 自动 embedding
- 自动 chunk 管理
- 向量检索
- rerank
- 多工作空间协同
- 工作空间级评测平台
- 完整导出体系
- 复杂权限系统

第一阶段的成功标准不是“能力多”，而是：

**让用户能在一个受控 Markdown 工作空间里，真正完成一次轻量知识处理闭环。**

## 产品判断

从当前产品形态看，这项能力是值得做的，原因不是“它能再做一次检索”，而是：

**它让用户在不进入正式 RAG 生命周期的前提下，也能对一组 Markdown 文档直接开展知识处理工作。**

这是一种明显不同于现有 RAG 的价值。

但它只在以下前提下成立：

- 明确是轻量模式
- 明确只处理 Markdown 文档集合
- 明确不复制一套完整 RAG
- 明确把价值放在分类、整理、验证，而不是再做一遍复杂检索系统
- 明确以受控命令行作为底座，而不是继续堆叠过早抽象的业务工具

## 最终建议

建议立项，但严格按“小而锋利”的方式推进。

适合的产品口径是：

> 在现有 RAG 之外，新增一种 Markdown 工作空间模式，让用户可以直接对文件夹中的 Markdown 文档执行轻量检索、阅读、分类整理与结果验证。

不适合的产品口径是：

> 再做一套轻量知识库 / 轻量 RAG / 文件夹版聊天系统。

前者会增强当前产品，后者会稀释当前产品。
