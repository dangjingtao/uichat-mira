# Markdown 工作空间能力评估

Status: Current
Owner: knowledge-base
Last verified: 2026-06-26
Layer: raw-source
Module: knowledge-base
Doc Type: design

面向当前产品形态，对“允许用户定义工作空间文件夹，并在 Markdown 文档集合上执行轻量检索、读取、分类识别、重建整理、验证”的能力做产品判断、边界定义与 MVP 收敛。

## 结论

这个能力值得做，但前提是把它定义为一个轻量文档工作模式，而不是第二套 RAG 系统。

当前项目已经有完整的 RAG 主链路。新增 Markdown 工作空间能力的价值，不在于再次提供知识库问答，而在于提供一种不需要入库、不需要分块、不需要 embedding，就能直接处理一组 Markdown 文档的工作方式。

## 建议的产品定义

建议把这项能力正式定义为：

**Markdown 工作空间模式**

特征如下：

- 用户手动指定一个文件夹作为工作空间
- 系统只处理其中的 Markdown 文档
- 不要求入库
- 不要求预分块
- 不要求 embedding
- 面向轻量知识处理，不替代现有 RAG

## 核心实现原则

这项能力的底座，不应该先定义为一组过早抽象的业务工具，而应该先定义为：

**让 AI 在用户指定的工作空间中，受控地直面命令行。**

原因如下：

- Markdown 文档集合本质上仍然是文件系统中的原始文件
- 检索、读取、批量处理、结构提取、验证，天然都可以通过命令行原语组合完成
- 命令行比过早封装的高层业务工具更轻、更通用、更接近真实工作流

## MVP 建议

如果只做最小可用范围，建议收敛为：

### 必做

- 工作空间文件夹选择
- 受控命令执行能力
- 命令输出与退出码采集
- 工作空间与路径边界控制
- Markdown 文档扫描
- 文本检索
- 文档全文/区块读取
- Markdown 结构解析
- 基于文档集合的分类识别
- 基于文档集合的整理输出
- 基于任务目标的结果验证

### 可延后

- 标签缓存
- 目录树视图增强
- 整理结果导出
- 工作区历史记录
- 与主聊天线程联动

## 产品判断

从当前产品形态看，这项能力是值得做的，原因不是“它能再做一次检索”，而是：

**它让用户在不进入正式 RAG 生命周期的前提下，也能对一组 Markdown 文档直接开展知识处理工作。**

这是一种明显不同于现有 RAG 的价值。

## 最终建议

建议立项，但严格按“小而锋利”的方式推进。

适合的产品口径是：

> 在现有 RAG 之外，新增一种 Markdown 工作空间模式，让用户可以直接对文件夹中的 Markdown 文档执行轻量检索、阅读、分类整理与结果验证。
