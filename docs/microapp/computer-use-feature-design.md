## 单点真相范围

这页只回答一件事：

在第一阶段只做浏览器场景的前提下，`computer_use` 微应用对用户到底长什么样、怎么用、遇到关键状态时怎么反馈。

它覆盖：

- 页面入口和信息架构
- 首次使用引导
- 浏览器运行时缺失时的安装引导
- 任务创建、审批、执行、回放、失败反馈
- MVP 非目标

它不覆盖：

- 具体接口字段
- 具体 Playwright 调用实现
- 宿主桌面自动化体验
- 浏览器插件接入体验

## 设计目标

第一阶段功能设计只服务三件事：

1. 用户能很快理解这不是聊天框，而是一个受控浏览器任务工作台。
2. 用户能明确知道任务现在卡在哪一步，是缺运行时、等审批、执行中，还是已经失败。
3. 用户能看见执行证据，而不是只收到一句“已经完成”。

## 页面定位

页面名称建议固定为：

- `Computer Use Studio`

它是一个独立微应用页面，不放进聊天消息流里伪装成对话附件。

第一阶段页面心智：

- 左侧输入目标和查看任务摘要
- 中间看浏览器执行状态
- 右侧看步骤、截图和结果回放

## 页面结构

### 1. 顶部栏

顶部栏只放高价值状态：

- 页面标题：`Computer Use Studio`
- 当前浏览器运行时状态
  - `Ready`
  - `Not Installed`
  - `Downloading`
  - `Broken`
- 当前任务状态
  - `Idle`
  - `Planning`
  - `Awaiting Approval`
  - `Running`
  - `Succeeded`
  - `Failed`
  - `Cancelled`

不要在顶部栏堆技术细节。

### 2. 左侧任务面板

左侧是“用户想做什么”。

建议包含：

- `Goal` 输入框
- 可选站点范围
  - 例如 `Only allow example.com`
- 风险提示摘要
  - 简短说明第一阶段只支持浏览器内操作
- 主按钮
  - `Create Plan`
  - `Start Task`
  - `Retry`

第一阶段不要做：

- 提示词工程工作台
- 多轮对话式编排
- 高级动作参数编辑器

### 3. 中间执行面板

中间区域是“当前到底在发生什么”。

第一阶段建议显示：

- 当前浏览器快照
- 当前步骤标题
- 当前步骤状态
- 是否暂停等待审批

如果没有任务在跑，就显示：

- 什么是 `Computer Use Studio`
- 当前仅支持浏览器场景
- 如果浏览器运行时没装，直接显示安装引导卡片

### 4. 右侧证据面板

右侧专门放证据，不和最终结果混成一块。

建议拆成三个 tab：

- `Plan`
- `Evidence`
- `Result`

这样用户更容易理解：

- 计划是什么
- 实际做了什么
- 最终得到了什么

## 首次使用流程

### 场景 A：浏览器运行时已就绪

1. 用户进入页面
2. 顶部显示 `Ready`
3. 左侧输入目标
4. 点击 `Create Plan`
5. 页面进入 `Planning`
6. 生成计划后，用户点击 `Start Task`

### 场景 B：浏览器运行时缺失

1. 用户进入页面
2. 顶部显示 `Not Installed`
3. 中间显示安装引导卡片
4. 用户点击 `Install Browser Runtime`
5. 页面显示下载进度
6. 完成后状态变成 `Ready`

第一阶段不要把这条流程包装成复杂的环境诊断向导。

## 运行时安装引导

第一阶段安装引导只回答三个问题：

1. 为什么现在不能开始任务
2. 需要安装什么
3. 安装完成后会发生什么

建议文案层只表达：

- 需要浏览器运行时才能执行浏览器任务
- 应用会优先复用已安装的运行时
- 如果本机没有可用浏览器运行时，可以由应用下载受管 Chromium

不要在用户第一眼就塞：

- Playwright 术语
- Node 术语
- Chromium 版本号
- 解压路径

这些信息可以放到“详情”抽屉里。

## 任务主流程

### 1. `Idle`

页面初始态。

用户能做的事：

- 输入目标
- 查看运行时是否可用
- 安装运行时

### 2. `Planning`

系统正在把目标转成执行计划。

用户看到：

- 计划生成中
- 暂不可开始执行

### 3. `Plan Ready`

系统已经生成第一版计划。

用户看到：

- 步骤列表
- 每一步的简短描述
- 哪些步骤可能需要确认

用户能做的事：

- `Start Task`
- `Edit Goal`
- `Discard`

第一阶段不做手工逐步改计划。

### 4. `Awaiting Approval`

执行卡在一个需要确认的步骤。

用户必须能看见：

- 当前要做什么动作
- 这个动作为什么需要确认
- 批准后会继续什么

用户能做的事：

- `Approve Once`
- `Cancel Task`

第一阶段不做：

- 永久批准
- 域名级批准策略编辑器
- 历史审批管理台

### 5. `Running`

系统正在执行浏览器动作。

用户必须能看见：

- 当前步骤
- 最新截图
- 最近几条动作记录

用户能做的事：

- `Stop Task`

第一阶段不做 `Pause / Resume`。

### 6. `Succeeded`

任务成功后，页面需要分清“过程”和“结果”。

结果区建议显示：

- 最终结论
- 关键截图
- 关键页面状态摘要
- 任务耗时

不要把所有动作日志都塞进结果卡片。

### 7. `Failed`

失败不是异常角落，而是主场景之一。

失败时页面必须说明：

- 失败在哪一步
- 当前能确认的原因
- 用户下一步可以做什么

建议按钮：

- `Retry`
- `Edit Goal`
- `View Evidence`

### 8. `Cancelled`

取消后要保留：

- 已执行到哪一步
- 为什么结束
- 已留下哪些证据

不要像没发生过一样直接清空。

## Plan / Evidence / Result 三块怎么分

### `Plan`

只放计划信息：

- 步骤顺序
- 风险标记
- 可能的审批节点

### `Evidence`

只放过程证据：

- 截图时间线
- 动作日志
- 页面摘要
- 失败快照

### `Result`

只放最终用户关心的产出：

- 最终结论
- 最终关键截图
- 可以复制的摘要

这三块必须分开，否则用户很快就会看不懂。

## 缺运行时、等审批、执行失败的反馈优先级

第一阶段最容易伤可用性的，不是动作能力少，而是状态说不清。

所以页面反馈优先级建议固定为：

1. 运行时不可用
2. 等待审批
3. 执行失败
4. 执行中进度
5. 成功结果

也就是说，页面最显眼的位置要优先告诉用户：

- 现在是不是根本跑不起来
- 现在是不是在等你点批准
- 现在是不是已经失败

不要只顾着做“成功时很唬人”的大图。

## MVP 非目标

第一阶段明确不做：

- 宿主桌面窗口操作
- 浏览器插件接管当前标签页
- 多浏览器切换
- 多任务并发工作台
- 聊天流内嵌执行
- 高级审批策略系统
- 录屏播放器
- 登录凭据托管

## 推荐页面文案方向

第一阶段文案要像受控工具，不要像万能代理。

适合的表达：

- `Create Plan`
- `Awaiting Approval`
- `Install Browser Runtime`
- `View Evidence`
- `Task Failed`

不适合的表达：

- `Let AI take over`
- `Run everything automatically`
- `One-click full desktop control`

## 功能分期建议

### Phase 1

浏览器内单任务工作台

- 单目标
- 单会话
- 单任务执行
- 单次审批
- 截图和结果回放

### Phase 2

任务历史和更完整的证据查看

- 历史任务列表
- 失败复看
- 更细的步骤详情

### Phase 3

再决定是否要扩到：

- 插件接入
- 聊天联动
- 宿主桌面能力

## 当前结论

第一阶段功能设计必须把这件事定义为：

- 一个浏览器任务工作台

不是：

- 一个会聊天的浏览器插件
- 一个宿主桌面遥控器
- 一个自动代打一切的黑箱

只要这条定义不漂，后面的实现边界就不容易走偏。
