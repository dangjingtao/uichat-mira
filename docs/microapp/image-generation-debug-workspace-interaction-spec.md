## 单点真相范围

这页只回答一件事：

如果要给 `image_generation` 微应用做第一版“仅供微应用界面调试”的桌面页，它的原型交互应该长什么样。

它覆盖：

- 页面目标
- 页面结构
- 关键交互流
- 状态反馈
- 设计语言和视觉约束

它不覆盖：

- backend 实现细节
- 最终像素稿
- chat、第三方平台或 MCP 暴露面

## 页面定位

这是一个调试工作区，不是画廊，也不是消费型创作社区。

它的核心任务只有三个：

1. 让用户快速发起一次生图任务
2. 让用户清楚看到当前任务处于什么状态
3. 让用户在成功或失败后拿到足够明确的结果和诊断信息

所以它的产品气质应该是：

- 专业
- 低噪音
- 可排查
- 可重复试验

而不是：

- 娱乐化
- 炫技型
- 社区型
- 作品集型

## 用户心智

用户进入这个页面时，默认带着下面两类目标之一：

### 1. 验证链路是否通

用户想知道：

- 这个 provider 能不能跑通
- 这个 prompt 能不能出图
- `ComfyUI workflow JSON` 能不能提交成功

### 2. 诊断问题在哪

用户想知道：

- 是参数写错了
- 是 provider 没响应
- 是任务还在排队
- 是远端成功了但本地产物没回收
- 是 `ComfyUI` workflow 本身有问题

所以页面不能只展示“生成中”或“失败了”。

页面必须让用户看得见：

- 当前模式
- 当前 provider
- 当前任务状态
- 当前请求摘要
- 当前结果或错误

## 页面总体结构

推荐采用双栏工作台布局。

```text
页面标题区
  -> 模式切换与环境说明

主工作区
  -> 左栏：请求配置与提交区
  -> 右栏：结果预览与执行状态区

底部附加区
  -> 请求摘要 / 执行日志 / 调试说明
```

## 页面信息架构

### 1. 顶部标题区

目标：

- 明确用户现在处于“生图调试工作区”
- 明确这不是 chat，也不是工具市场
- 明确当前任务只服务微应用界面调试

应包含：

- 页面标题
  - `Image Generation Studio`
- 页面副标题
  - 一句话说明这是用于调试 `provider` 和 `workflow` 的微应用工作区
- 当前限制提示
  - 例如：
  - `当前仅支持微应用界面内调试，不会同步接入 chat 或第三方入口`

可选辅助信息：

- 当前默认 provider
- 当前运行模式说明

### 2. 左栏：请求配置区

左栏是操作主区。

它建议再分成三个分组卡片。

#### A. 模式与 Provider 卡

目标：

- 先让用户明确当前在用哪种执行模式

应包含：

- 执行模式切换
  - `Prompt 模式`
  - `Workflow 模式`
- Provider 选择
  - 首批只需要展示：
  - `OpenAI Images`
  - `阿里云万相`
  - `腾讯云混元`
  - `ComfyUI Local`
- Model 字段
  - 如果当前 provider 需要显式 model，则展示可编辑输入或下拉
- 环境状态提示
  - 例如：
  - `云 provider 已配置`
  - `ComfyUI endpoint 未连接`
  - `当前 provider 需要 API Key`

交互规则：

- 当用户切到 `Workflow 模式` 时，provider 自动切换到 `ComfyUI Local`，或只保留 `ComfyUI Local` 可选
- 当用户切回 `Prompt 模式` 时，workflow 编辑区折叠并清空“当前提交态”
- 如果 provider 不支持当前模式，对应选项不可用，而不是切换后再报错

#### B. Prompt 请求卡

目标：

- 让用户快速组织一次 prompt 型任务

仅在 `Prompt 模式` 下显示。

应包含：

- `Prompt` 多行输入框
- `Negative Prompt` 多行输入框
- `尺寸 Size` 选择器
- `风格 Style Preset` 选择器
- `张数 Count`
  - 第一版建议固定显示为 `1`
- 高级参数折叠区
  - `Seed`
  - Provider-specific 参数入口

交互规则：

- `Prompt` 是主输入，视觉层级最高
- `Negative Prompt` 默认弱一层，不应和主 prompt 抢主次
- `高级参数` 默认折叠，避免新用户一上来被技术字段淹没
- Provider-specific 参数不要直接散在主表单中，应放在“高级参数”区域

#### C. Workflow 请求卡

目标：

- 让用户提交一份 `ComfyUI workflow API JSON`

仅在 `Workflow 模式` 下显示。

应包含：

- `Workflow JSON` 大文本编辑区
- `上传 JSON 文件` 按钮
- `运行时替换参数` 小区域
  - 例如：
  - prompt 覆盖值
  - seed 覆盖值
- `JSON 格式状态`
  - `格式合法`
  - `不是合法 JSON`
  - `不是 ComfyUI API format`

交互规则：

- 文本编辑和文件上传是同一数据源，不是两套互不相干入口
- 上传成功后，编辑区内容同步刷新
- 如果 JSON 非法，主提交按钮不可用
- 如果 JSON 合法但不是 API format，应给明确错误，不要只说“提交失败”

#### D. 提交动作区

目标：

- 让用户明确什么时候可以提交、当前是否正在执行、是否允许重试

应包含：

- 主按钮
  - `开始生成`
- 次按钮
  - `重置输入`
- 任务状态提示
  - 例如：
  - `当前无运行中任务`
  - `任务排队中`
  - `任务执行中`

交互规则：

- 有运行中任务时，主按钮改为 disabled 或 loading
- 第一版不做“同时多个任务并发提交”
- 如果用户修改了输入但尚未重新提交，要有“表单已变更”的轻提示

### 3. 右栏：结果与状态区

右栏是反馈主区。

它建议分成三个层级。

#### A. 结果预览卡

目标：

- 第一眼就告诉用户这次有没有出图

状态分支：

- 空态
  - 还没有任何结果
- 生成中
  - 显示大预览骨架屏或占位图
- 成功
  - 显示图片主预览
- 失败
  - 显示失败态面板，而不是空白

成功态应包含：

- 图片主预览
- 图片尺寸
- 产物来源标签
  - `base64`
  - `remote-url 已回收`
  - `local-file`
- 本次生成时间

失败态应包含：

- 明确错误标题
  - 例如：
  - `任务执行失败`
- 简短错误摘要
- 一个“查看详细诊断”的次级动作

#### B. 任务状态卡

目标：

- 把“后台到底在干什么”讲清楚

应包含统一状态轴：

- `queued`
- `running`
- `succeeded`
- `failed`
- `cancelled`
- `blocked`

展示方式建议：

- 垂直步骤状态条，或单行状态时间轴

每个状态下的用户感知文案建议：

- `queued`
  - 已提交，等待 provider 或本地 runner 开始执行
- `running`
  - 正在生成或正在等待远端结果返回
- `succeeded`
  - 图片已成功生成并回收到本地结果区
- `failed`
  - 任务执行失败，未得到有效图片结果
- `cancelled`
  - 当前任务已被用户取消或被系统主动停止
- `blocked`
  - 当前任务因为配置、权限或环境问题无法继续执行

#### C. 请求摘要与调试摘要卡

目标：

- 给用户一个快速复盘面

应包含：

- 本次 provider
- 本次 model
- 本次模式
- prompt 摘要或 workflow 摘要
- size / style / seed
- provider job id
  - 如果有
- 本地产物 id
  - 如果有

设计要求：

- 这是调试信息，不是正文内容
- 视觉上应使用中性面、等宽片段或结构化列表
- 不要把它做成大段原始 JSON 直接扑到页面上

### 4. 底部附加区

底部区只在有价值时展开。

建议包含两个折叠区：

#### A. 执行日志

目标：

- 帮用户定位是卡在“提交前、排队中、执行中、回收中”的哪一步

可展示：

- 提交时间
- 状态更新时间
- 当前阶段
- 简短事件列表

#### B. 使用说明

目标：

- 给第一次进来的用户低成本提示

应包含：

- `Prompt 模式` 适合什么场景
- `Workflow 模式` 适合什么场景
- `ComfyUI JSON` 必须是 API format
- 当前页面只服务微应用界面调试

## 关键交互流

### 流 1：Prompt 模式成功链路

1. 用户进入页面
2. 页面默认落在 `Prompt 模式`
3. 用户选择 provider
4. 用户填写 `Prompt`
5. 用户按需选择尺寸和风格
6. 用户点击 `开始生成`
7. 左栏进入提交中态
8. 右栏状态卡从 `queued` 进入 `running`
9. 右栏预览卡显示加载占位
10. 生成成功后，右栏展示主图和请求摘要

设计重点：

- 提交后不要让用户失去对当前输入的可见性
- 结果回来之前，右栏必须明确告诉用户“任务还活着”

### 流 2：Prompt 模式失败链路

1. 用户提交任务
2. 任务进入 `queued` 或 `running`
3. provider 返回失败
4. 状态卡切到 `failed`
5. 结果卡显示失败摘要
6. 调试摘要卡显示 provider / model / request id / 错误信息

设计重点：

- 失败态不能退化成空白页
- 失败后用户应保留上一次输入，方便直接修正后重试

### 流 3：Workflow 模式成功链路

1. 用户切换到 `Workflow 模式`
2. 页面自动把 provider 锁定或收敛到 `ComfyUI Local`
3. 用户粘贴或上传 workflow JSON
4. 页面先做 JSON 合法性检查
5. 用户点击 `开始生成`
6. 状态卡展示 `queued -> running -> succeeded`
7. 结果卡展示图片预览
8. 摘要卡显示本次 workflow 运行信息

设计重点：

- 用户必须先被明确提示“这里只接受 API format workflow JSON”
- 不要让用户以为任意 ComfyUI 导出 JSON 都能直接运行

### 流 4：Workflow 模式输入不合法

1. 用户切换到 `Workflow 模式`
2. 用户上传错误 JSON 或粘贴无效内容
3. 页面立即在工作流卡里给出格式错误提示
4. 主提交按钮保持不可用

设计重点：

- 错误要就地展示在 workflow 卡内
- 不要等用户点击提交后再报“未知错误”

## 状态矩阵

### 页面级状态

- `initial-loading`
  - 页面首次加载 provider 信息和默认配置
- `ready`
  - 表单可编辑
- `submitting`
  - 已发起一次任务提交
- `polling`
  - 正在等待任务结果
- `terminal-success`
  - 有可展示结果
- `terminal-failed`
  - 有明确失败结果

### 表单级状态

- `clean`
  - 当前输入未改动
- `dirty`
  - 当前输入已改动但未提交
- `invalid`
  - 输入不满足最小要求
- `locked-by-running-job`
  - 当前任务运行中，表单部分受限

### 结果级状态

- `empty`
- `preview-loading`
- `preview-ready`
- `preview-failed`

## 关键组件说明

### 模式切换器

建议使用清晰的分段切换，而不是下拉框。

原因：

- 用户需要明显感知自己当前在 `Prompt` 还是 `Workflow`
- 这会直接改变整块表单结构

### Provider 选择器

建议使用带说明的下拉，而不是只列 provider 名字。

每个选项建议带一句辅助描述：

- `OpenAI Images`
  - 国际直连，适合快速验证 prompt 生图
- `阿里云万相`
  - 国内云 provider，适合验证国内链路
- `腾讯云混元`
  - 异步任务型接口样板
- `ComfyUI Local`
  - 本地工作流执行器

### Prompt 输入框

建议是页面里视觉最强的输入区。

要求：

- 默认显示 5 到 7 行高度
- 支持长文本滚动
- placeholder 不要写成空泛鸡汤，应该提示用户写具体图像描述

### Workflow 编辑区

建议使用明显更偏“技术调试”的视觉语言：

- 等宽字体
- 中性深浅分层
- 明确的格式状态提示

但不要做成黑底终端感过强的编辑器。

### 结果预览区

结果区要让图片本身成为主角，但仍然保持工具页气质。

要求：

- 有稳定比例的预览框
- 大图显示优先
- 失败态也占据同等重要区域，不要缩成一条小红字

## 反馈与文案语气

### 成功反馈

语气应：

- 明确
- 克制
- 不夸张

推荐：

- `图片已生成`
- `结果已回收到本地预览区`

不推荐：

- `太棒了！你的杰作诞生了`
- `AI 灵感已爆发`

### 失败反馈

语气应：

- 直说问题
- 方便继续排查

推荐：

- `任务执行失败`
- `当前 provider 未返回有效图片结果`
- `Workflow JSON 不是可执行的 API format`

不推荐：

- `生成过程有点小问题`
- `系统正在努力恢复`

### 运行中反馈

语气应：

- 强调阶段
- 不假装快结束

推荐：

- `任务已提交，等待开始执行`
- `正在生成图片`
- `正在回收远端结果`

不推荐：

- `马上就好`
- `即将完成`

## 视觉与布局约束

这页应遵守当前 UI 设计规范：

- 紧凑
- 信息层级清楚
- 不用夸张阴影和大圆角
- 主要依赖语义 token 做层次

建议：

- 主背景使用 `surface-*` 体系
- 状态提示使用 `success-soft / warning-soft / danger-soft / info-soft`
- 卡片之间通过结构和留白区分，而不是靠花哨渐变

不建议：

- 大面积发光背景
- 全页插画压主信息
- 把状态栏做成娱乐化 AI 仪表盘

## 交付给设计师的明确要求

设计输出至少应包含：

1. 页面整体布局稿
2. `Prompt 模式` 完整态
3. `Workflow 模式` 完整态
4. `初始空态`
5. `生成中态`
6. `成功态`
7. `失败态`
8. `Workflow JSON 非法态`
9. 响应式收缩方案
  - 至少说明窄宽度下左右栏如何重排

## 设计验收检查单

设计稿出来后，应逐条检查：

1. 用户是否一眼知道这是调试工作区，而不是聊天页或作品页。
2. `Prompt` 和 `Workflow` 两种模式是否足够清晰地区分开。
3. 运行中、成功、失败三种状态是否一眼可辨。
4. 失败时是否仍然保留足够诊断信息，而不是只有一句报错。
5. `ComfyUI` 的 workflow 入口是否足够明确地提示了 API format 约束。
6. 页面是否仍然保持专业、克制、可排查，而不是被“AI 炫技感”带偏。
