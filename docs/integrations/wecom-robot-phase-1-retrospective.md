## 背景

这次改造的起点，不是为了“多接一个第三方平台”，而是为了验证：

- 当前本地桌面应用，能不能把企业微信作为真实可用的企业入口
- 企业微信能不能承接我们现有 Chat / RAG / 知识库能力
- 在没有稳定公网回调条件的前提下，什么接法最现实

第一期真正想解决的问题其实很朴素：

- 能不能让用户在企业微信里 `@机器人`
- 机器人能不能调用我们本地 RAG
- 最终把答案稳定回到企业微信里

## 第一阶段目标

第一期的目标后来被收敛成三件事：

1. 跑通企业微信智能机器人长连接
2. 让机器人能够调用我们本地知识库 RAG
3. 在项目里补齐最小配置入口、状态显示和测试能力

这里有一个很重要的收敛：

- 第一阶段主线不是网页授权
- 也不是自建应用全能力接入
- 更不是复杂的组织权限引擎
- 而是“企业微信机器人作为对话入口”的最小闭环

## 最终做成了什么

第一期最核心的成果，是把下面这条链路真正跑通了：

```text
企业微信里 @ 智能机器人
-> 本地服务收到消息
-> 提取问题
-> 选择知识库
-> 调用现有 RAG pipeline
-> 返回结果给企业微信机器人
```

围绕这条主线，第一期已经完成了这些落地：

- 企业微信设置页收口
- 智能机器人 `Bot ID / Secret` 配置保存
- 知识库选择配置
- 机器人连接状态显示
- 最小测试消息发送能力
- 后端智能机器人长连接接入
- 本地 RAG 适配器打通
- 企业微信群机器人 webhook 测试链路

从结果看，这说明一件事：

- “本地桌面应用 + 企业微信智能机器人 + 本地 RAG” 这条方向是成立的

## 做对了什么

### 1. 选对了第一主线

这次最正确的决策，是把主线从“网页授权 / 回调域名”切到了“智能机器人长连接”。

原因很直接：

- 我们是本地应用
- 没有天然公网回调能力
- 回调域名、备案、国内可达性都带来很高摩擦

而智能机器人长连接模式恰好反过来：

- 不依赖公网回调
- 更贴合桌面端常驻进程
- 更容易先验证真实对话价值

这一步决定，直接避免了第一期陷入基础设施泥潭。

### 2. 复用了现有 RAG 主体

我们没有新做一套“企微专用问答引擎”，而是复用现有知识库和 RAG pipeline。

这件事的价值非常大：

- 机器人不是孤立能力
- 知识库选择逻辑可以直接沿用
- 后面 Chat 主界面和企业微信入口能共享同一套回答能力

这让企业微信接入成为“现有能力的入口扩展”，而不是新的平行系统。

### 3. 前端收得住

第一期没有新开大面板，而是先把能力收口到设置页。

这个取舍是对的，因为此时真正缺的是：

- 配置入口
- 状态可见性
- 最小测试闭环

而不是新的复杂运营后台。

### 4. 尽早识别出两类机器人不是一回事

改造过程中，一个很重要的认识逐渐清晰：

- 群机器人 webhook 适合做通知出口
- 智能机器人长连接适合做对话入口

这两个能力虽然都叫“机器人”，但不是一条链。

这个边界一旦明确，后面的设计就清楚了很多：

- webhook 负责发通知
- 智能机器人负责收问答

## 踩过的坑

### 1. 一开始探索面过宽

第一期前半段同时碰了很多路线：

- 自建应用
- 网页授权
- Cloudflare Worker 回调
- 群机器人 webhook
- 智能机器人长连接

这导致阶段目标一度发散，判断成本很高。

复盘来看，第一期应该更早明确：

- 智能机器人长连接是主线
- 网页授权保留为延后实现
- webhook 只做通知补充

### 2. 把“看不到回复”过早等同于“RAG 挂了”

这是本次调试里最典型的误判来源。

后面的日志已经说明：

- 很多场景下消息已经收到
- RAG 已经调用
- 甚至 `RAG completed` 已经出现

但企业微信前端仍然不稳定展示最终回复。

也就是说，问题并不总在：

- rewrite
- retrieve
- rerank
- generate

而常常落在最后一跳：

- 智能机器人回复协议
- SDK 调用方式
- 企业微信客户端展示行为

这个教训很重要：

- 第三方机器人问题必须按链路拆层排查
- “没看到结果”不能直接推导成“RAG 没出答案”

### 3. 过早尝试加记忆

在主链路还没有完全稳定的时候，尝试引入多轮记忆，会同时放大这些不确定性：

- 提问归一化
- 会话隔离
- TTL
- 上下文拼接
- RAG 重写行为

这在阶段上是超前的。

复盘结论很明确：

- 第一阶段应该坚持单轮
- 先稳定收发
- 再谈多轮体验

### 4. 配置概念容易混淆

这次前后反复暴露出一个产品问题：

- `Robot Webhook URL`
- `智能机器人 Bot ID`
- `智能机器人 Secret`
- `自建应用 AgentID / Secret`

这几组配置来自不同能力线，但在用户视角里很容易混成一组。

这导致的直接后果是：

- 报错看起来像“企微不稳定”
- 实际上是配置项填到了错误的能力线上

这说明设置页和文档都需要把能力用途写得更清楚。

## 本次最重要的判断收敛

经过第一期，企业微信这条业务线的定位已经比较清楚了。

### 1. 智能机器人长连接

定位：

- 对话入口
- 企业微信里 `@机器人` 的 RAG 问答入口

### 2. 群机器人 webhook

定位：

- 通知出口
- 测试消息、任务结果、导入完成、失败告警等结果推送

### 3. 自建应用 / 网页授权

定位：

- 身份绑定和组织能力扩展的后续基础设施
- 当前保留设计，不作为第一期主链依赖

也就是说，第一期之后最合理的能力分层是：

```text
企业微信智能机器人 -> 对话入口
企业微信群机器人 webhook -> 通知出口
企业微信自建应用 / OAuth -> 后续身份与组织扩展
```

## 第一阶段的真实得失

### 得

- 证明了本地应用可以直接接企业微信智能机器人
- 证明了企业微信可以复用现有知识库与 RAG
- 证明了设置页收口这条前端策略可行
- 明确了通知与对话应走不同机器人能力
- 为后续 Chat 主界面插件化接入提供了实际基础

### 失

- 前期探索路线过多，导致推进成本偏高
- 机器人回复展示协议问题比预期更难缠
- 一度把 RAG 问题和回复协议问题混在一起
- 引入过早的记忆尝试，增加了噪声
- 配置项命名和用途区分仍不够清晰

## 第一阶段的核心结论

这次改造最大的结论不是“企微接上了”，而是：

- 企业微信可以成为我们现有 RAG 系统的企业入口层

它不是一个独立子产品，也不是一个只停留在设置页的接入项。

更准确的定位应该是：

- Chat / RAG / 知识库能力的企业微信入口

而从工程结构上看，后续应该坚持：

- backend-first
- 配置收口在 Settings
- 消费入口回到 Chat 主界面
- 但通过插件或工具能力接入，而不是直接把企微协议塞进 chat runtime

## 第二阶段建议

第一期之后，建议不要再继续横向发散，而是沿着主线纵向做稳。

优先级建议如下：

### 1. 先稳定智能机器人回复

目标：

- 确认企业微信客户端稳定展示回复的协议写法
- 让“RAG completed 就能稳定看到结果”成为确定行为

这是当前最高优先级。

### 2. 继续保留单轮

目标：

- 不再引入会话记忆
- 不再加多轮上下文拼接
- 先把单轮准确性、稳定性、可观测性做扎实

### 3. 把配置与用途区分写清楚

目标：

- 在设置页明确区分：
  - webhook 用于通知
  - smart robot 用于对话
  - 自建应用用于后续身份 / 组织扩展

### 4. 把消费入口逐步收回 Chat

目标：

- 企业微信能力最终不只停留在“设置页可配置”
- 而要作为插件能力回到 Chat 主界面被消费

这也是后面做自家 MCP 包装的现实基础。

## Recommendation

如果只保留一句话作为第一期复盘结论，那就是：

- 第一阶段已经证明“企业微信智能机器人 + 本地 RAG”这条路线值得继续做，但下一阶段必须停止发散，围绕“稳定回复展示”这一个关键问题继续收敛。

在产品上，要把企业微信看成我们现有系统的企业入口层。

在工程上，要坚持：

- webhook 做通知
- 智能机器人做对话
- OAuth / 自建应用保留为后续增强

这是目前最符合当前项目形态、也最不容易继续失焦的路线。
