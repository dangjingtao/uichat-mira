## 单点真相范围

这页只回答一件事：

如果当前项目正式开始设计飞书接入，我们系统到底要做什么、不做什么，以及一期完整交付如何落地。

它覆盖：

- 飞书接入的系统定位
- 我们真正要交付的能力边界
- 一期完整交付范围
- backend / frontend / 知识库链路的落点
- 与企业微信的角色差异

它不覆盖：

- 飞书全部开放 API 清单
- 某个单点接口的代码实现细节
- 最终生产级组织治理、审计与运维方案

## 1. 设计目标

飞书接入不是为了“多接一个企业平台”。

从当前系统视角看，飞书接入的目标应该是：

1. 让飞书中的消息入口能够调用我们自己的知识库能力
2. 让飞书成为知识库结果的通知出口之一
3. 把组织身份、权限收口和知识源扩展一起纳入一期
4. 保持 backend-first，不让飞书协议进入业务主链路

这里必须提前写清楚：

- 飞书接入的核心不是机器人本身
- 机器人的价值只是把外部平台流量导入我们的知识库问答链路

一句话说：

> 飞书是知识库能力的企业入口层，不是系统本体。

## 2. 系统定位

当前项目里的飞书，应被定义为三类能力的组合：

### 2.1 问答入口

含义：

- 飞书机器人收到用户消息
- 把问题送入我们本地知识库问答链路
- 再把结果返回给飞书

这条链路的本质是：

```text
飞书消息 -> 本地知识库调用 -> 飞书回复
```

### 2.2 通知出口

含义：

- 知识库导入完成
- 任务执行完成
- 评测结果产出
- 失败告警

这类结果可以主动回推到飞书用户或群。

### 2.3 知识源接入

这是飞书相对企微更值得落地的核心优势之一。

含义：

- 飞书文档、沉淀材料、FAQ 文稿等，可作为知识库导入源

### 2.4 身份与组织

含义：

- 飞书用户与本地账户绑定
- 飞书组织结构进入本地组织模型
- 后续可直接服务于知识库 ACL、资源授权和用户画像

### 2.5 消息归档与检索

含义：

- 选定范围内的飞书消息进入本地归档
- 归档消息建立全文索引
- 后续可被检索、分析或转化为知识资产

## 3. 我们要做什么

飞书一期，系统要完成下面六件事：

### 3.1 让飞书机器人调用知识库

这是第一主线。

交付目标：

- 配置飞书消息入口
- 收到飞书消息
- 抽取用户问题
- 调用现有知识库 / RAG pipeline
- 返回一条稳定回复

注意：

- 外部平台只负责“带来问题”
- 回答能力仍然属于我们自己的知识库系统

### 3.2 让飞书承担通知出口

交付目标：

- 至少支持一条主动推送链路
- 推荐优先做：
  - 知识库导入完成通知
  - 任务失败通知

### 3.3 预留身份与组织扩展位
一期直接交付：

- 身份绑定主链
- 组织同步主链
- ACL 对接位

### 3.4 让飞书文档进入知识库

交付目标：

- 能选择飞书文档
- 能进入现有知识库导入链路
- 能形成可追踪的导入结果

### 3.5 让飞书消息进入归档与检索

交付目标：

- 能归档选定范围的飞书消息
- 能建立全文索引
- 能支持后续查询与知识沉淀

### 3.6 保持配置收口

飞书接入仍然应通过：

- `Settings -> 企业集成`

完成配置和状态收口。

不新开复杂管理后台。

## 4. 我们不做什么

第一阶段明确不做下面这些：

### 4.1 不做完整飞书协作平台替代

不做：

- 飞书侧完整业务前端嵌入
- 把飞书当作系统主界面

### 4.2 不做复杂平台治理

不做：

- 多租户平台运营台
- 大规模实例运维界面
- 审批中心
- 复杂组织策略编排

### 4.3 不做“飞书替代 Chat 主界面”

这点必须明确：

- 飞书只是知识库问答入口之一
- 不是唯一主交互面
- 本地 Chat 主界面仍然是我们最可控、最完整的消费入口

### 4.4 不做跨平台超级抽象

虽然底层已经是 `Provider -> Instance -> Capability`，
但第一阶段不应该为了飞书去做过度抽象。

我们只需要：

- 在统一模型里接一个 `lark` provider
- 接必要 capability
- 不把平台共性抽象过头

## 5. 第一阶段推荐能力边界

从系统设计上，飞书一期收敛到：

- `lark.bot`
- `lark.webhook`
- `lark.knowledge_source`
- `lark.identity`
- `lark.org_sync`
- `message_archive`
- `message_search`

这里的优先级建议是：

### 第一优先

- `lark.bot`

原因：

- 它能直接验证“飞书消息是否能调用知识库”
- 这是飞书接入的核心主线

### 第二优先

- `lark.webhook`

原因：

- 便于把通知回推链路单独稳定
- 与问答入口职责分离

### 第三优先

- `lark.knowledge_source`

原因：

- 这是飞书和知识库结合的关键能力

### 第四优先

- `lark.identity`
- `lark.org_sync`

原因：

- 身份和组织是知识库权限、用户识别和资源授权的基础

### 第五优先

- `message_archive`
- `message_search`

原因：

- 这是把飞书会话沉淀为长期知识资产的关键一步

## 6. 飞书与企业微信的角色差异

从当前系统看，两者不是简单平替关系。

### 企业微信更像

- 组织治理型入口
- 通知与企业内协作入口
- 偏“权限、通知、组织”语义

### 飞书更像

- 知识协作型入口
- 问答与文档知识延展入口
- 偏“知识、文档、协作”语义

但当前阶段不要把这个差异夸大成两套系统。

统一结论是：

- 两者都先做消息入口 + 通知出口
- 飞书一期额外纳入知识源、身份组织、归档检索

## 7. 系统落点

### 7.1 Backend

backend 负责：

- 飞书配置读取
- token / secret 管理
- 机器人消息接入
- 知识库调用编排
- 通知发送
- 知识源导入接口
- 身份绑定接口
- 组织同步接口
- 消息归档与全文索引接口

建议目录延续现有设计：

```text
server/src/integrations/lark/
  config.ts
  client.ts
  provider.ts
  bot.adapter.ts
  webhook.adapter.ts
  knowledge-source.adapter.ts
  identity.adapter.ts
  org-sync.adapter.ts
  message-archive.adapter.ts
```

### 7.2 Frontend

frontend 只负责：

- 展示飞书 provider
- 展示当前实例
- 展示 bot / webhook / knowledge_source / identity / org_sync 配置入口
- 展示状态和各能力测试入口

不负责：

- 直接调用飞书 API
- 保存 secret
- 理解飞书协议细节

### 7.3 知识库主链路

飞书接入后，知识库仍然是主系统真相。

也就是说：

- 飞书不会拥有一套独立问答引擎
- 飞书也不会拥有一套独立知识库存储
- 它只是调用我们现有的知识库和 RAG pipeline

这条原则必须保持不变。

## 8. 一期完整闭环

飞书一期至少要完成下面这些闭环：

```text
飞书机器人消息
-> 本地服务收到消息
-> 提取问题
-> 调用现有知识库
-> 返回答案到飞书
```

第二条：

```text
知识库导入完成
-> backend 产生业务事件
-> 飞书 webhook / 消息发送
-> 飞书收到通知
```

第三条：

```text
飞书文档
-> 本地服务拉取内容
-> 进入知识库导入链路
-> 导入结果可见
```

第四条：

```text
飞书身份 / 组织
-> 本地绑定与同步
-> 进入本地用户 / ACL 体系
```

第五条：

```text
飞书消息
-> 本地归档
-> 全文索引
-> 可被检索
```

这些闭环完成后，飞书一期才算成立。

## 9. 已知边界

飞书这条线也要提前写清楚边界，避免后续误判。

### 9.1 平台入口不等于系统入口

飞书只是外部入口，不是系统本体。

### 9.2 第三方平台不在我们完全控制范围内

无论是消息模型、审核规则、平台治理还是消息投递策略，
都不归我们完全控制。

所以：

- 我们只对“平台已投递到本地服务的消息”负责
- 平台侧未进入本地服务的部分，不在我们控制范围内

### 9.3 无公网域名仍是硬约束

飞书一期虽然扩大了范围，但接法仍要服从一个约束：

- 默认不依赖公网回调域名

这意味着：

- 机器人消息入口优先走长连接
- 某些身份 / 登录官方路径如果强依赖回调域名，需要单独设计替代接法

## 10. 设计结论

飞书接入在当前系统里的正确边界应定义为：

> 以知识库调用为核心，同时把消息入口、通知出口、知识源接入、身份组织同步、消息归档检索一起纳入一期交付。

更直白一点：

- 我们不是在做“飞书接入本身”
- 我们是在做“知识库能力的飞书企业入口化与知识沉淀化”

这就是飞书设计的起点，也是一期完整交付不应偏离的边界。
