# 轻量桌面版企业集成前端系统设计

## 1. 设计定位

这是一个`本地桌面应用中的个人接入面板`，不是 SaaS 平台后台。

因此前端目标不是：

1. 管理很多企业
2. 监控很多机器人
3. 做多租户运营
4. 承载复杂平台治理

前端目标是：

1. 让当前用户快速接入一个企业平台
2. 快速知道“现在能不能用”
3. 快速修改少量关键配置
4. 快速测试一条链路
5. 把复杂性留在后端模型里，不暴露给用户

一句话定义：

> 前端是“我的企业接入工具”，不是“企业平台控制台”。

---

## 2. 前后端职责分离

### 2.1 后端职责

后端继续保留通用模型，支持未来扩展：

1. 平台
2. 实例
3. 能力
4. 配置
5. 状态
6. 生命周期

后端可以按平台化方式设计，因为这是系统能力基础。

### 2.2 前端职责

前端只表达“当前用户最关心的操作对象”，不完整映射后端所有抽象。

前端只需要让用户理解：

1. 我接的是哪个平台
2. 当前接入是否可用
3. 我主要在用哪种方式
4. 如何修改配置
5. 如何测试

也就是说：

> 后端可以通用化，前端必须个人化、轻量化。

---

## 3. 前端核心对象模型

前端不直接把“实例 + 能力”并列展示给用户，而是做一层收口。

推荐前端对象模型如下：

### 3.1 平台

当前支持：

1. 企业微信
2. 飞书（未来）
3. 钉钉（未来）

平台只负责切换上下文。

### 3.2 当前接入

前端主对象不是“实例列表”，而是`当前接入`。

当前接入的语义是：

1. 当前这个平台下，用户正在使用的那一个接入单元

它在后端可能对应一个 instance，
但前端不需要强调“instance”这个抽象。

### 3.3 接入方式

接入方式是当前接入下的功能类型，例如：

1. Webhook 机器人
2. 智能机器人
3. OAuth 身份绑定
4. 通讯录同步

这里不要叫“能力”也行，可以叫：

1. 接入方式
2. 功能模块
3. 可用方式

对普通用户更容易理解。

---

## 4. 页面结构设计

页面名：`企业集成`

页面只保留三层结构：

1. 页面头部
2. 当前平台内容区
3. 配置抽屉 / 配置弹窗

不做多层嵌套，不做卡片墙，不做运维台。

---

## 5. 页面区块说明

### 5.1 页面头部

包含：

1. 小标题：企业集成
2. 主标题：企业集成
3. 一段短说明
4. 右侧操作按钮

说明文案建议：

> 管理本地企业平台接入。完成配置后，可在聊天或自动化流程中使用。

按钮建议只保留：

1. 刷新
2. 配置当前接入

不建议默认出现：

1. 新建很多实例
2. 批量管理
3. 复杂操作入口

### 5.2 平台切换区

形式：Tab

当前：

1. 企业微信

未来：

1. 飞书
2. 钉钉

Tab 只做平台切换，不承载统计面板，不承载介绍卡片。

### 5.3 当前接入总览区

这是页面主体。

页面不展示“很多实例卡片”，而只展示一个`当前接入卡`。

卡片建议展示：

1. 平台名称
2. 当前接入名称
3. 当前接入状态
4. 当前主接入方式
5. 最近错误摘要
6. 配置入口

例如：

1. 企业微信
2. 当前接入：默认企业微信
3. 状态：可用 / 待配置 / 异常
4. 接入方式：智能机器人 / Webhook 机器人
5. 最近状态：已连接 / 未配置 / 启动失败

这张卡片的职责是：

> 告诉用户“我现在是不是已经接好了”。

### 5.4 接入方式区

在当前接入卡下面，不再展示很多机器人卡片，而是展示一个轻量列表。

建议形式：

1. 单列列表
2. 每行一个方式
3. 每行只放摘要

每个接入方式展示：

1. 名称
2. 状态
3. 简短说明
4. 操作入口

例如：

1. 智能机器人
2. Webhook 机器人
3. OAuth 身份绑定

不要直接在这里摊开表单。

---

## 6. 交互路径设计

### 6.1 主路径

主路径应该非常短：

1. 用户进入企业集成页
2. 看到当前平台
3. 看到当前接入状态
4. 选择一个接入方式
5. 打开配置抽屉
6. 修改字段
7. 保存
8. 测试
9. 得到结果

这就是完整闭环。

### 6.2 配置方式

推荐使用：

1. 右侧抽屉

不建议：

1. 再跳二级页面
2. 多层弹窗嵌套
3. 页面内直接展开成长表单

原因：

1. 桌面应用讲究短路径
2. 用户更像“顺手改一下”
3. 配置完成后要快速回到概览

---

## 7. 配置抽屉设计

抽屉中只保留三块：

1. 基础配置
2. 接入方式配置
3. 调试结果

### 7.1 基础配置

包含：

1. 接入名称
2. 企业标识
3. 启用状态
4. 平台类型只读

这部分解决：

1. 当前接入是谁
2. 是否启用

不放复杂 JSON。

### 7.2 接入方式配置

这是最重要的部分。

每种接入方式只显示结构化表单，不允许 JSON 直配。

#### 智能机器人

字段建议：

1. Bot ID
2. Secret
3. 绑定知识库
4. 回复模式
5. 启用开关

#### Webhook 机器人

字段建议：

1. Webhook URL
2. Webhook Secret
3. 启用开关

#### OAuth 身份绑定

字段建议：

1. CorpID
2. AgentID
3. Secret
4. 回调说明
5. 当前状态

原则：

1. 所有字段表单化
2. `secret` 默认不回显
3. 不允许在主界面明文展示机密字段
4. 不允许直接展示整段配置 JSON

### 7.3 调试结果

调试区只回答三个问题：

1. 能不能连通
2. 错在哪
3. 下一步怎么做

展示内容建议：

1. 当前状态
2. 最近一次错误
3. 最近一次成功时间
4. 一个测试按钮

注意：
不同接入方式的测试动作不同。

#### 智能机器人

显示：

1. 启动
2. 停止
3. 当前连接状态

不显示“发送测试消息”。

#### Webhook 机器人

显示：

1. 测试消息内容
2. 消息格式
3. 发送测试消息

不显示长连接状态按钮。

也就是说：

> 调试区按接入方式变化，而不是强行统一一个测试按钮。

---

## 8. 设计原则

### 8.1 轻量原则

前端默认只服务“当前用户、当前平台、当前接入”。

### 8.2 去平台化外观

虽然系统底层是平台化的，但前端不能长成 SaaS 控制台。

### 8.3 去运维化表达

不要出现太多监控、统计、运行面板、运营字段。

### 8.4 表单化优先

优先结构化字段，不允许把 JSON 当主要配置方式。

### 8.5 机密字段保护

所有 secret 类字段：

1. 默认不回显
2. 可重新输入
3. 不直接展示历史值

### 8.6 单主对象原则

主页面始终围绕“当前接入”展开，不做实例墙。

---

## 9. 明确不做的内容

这版前端不做：

1. 多企业运营看板
2. 多实例矩阵管理
3. 大量机器人并列展示
4. 复杂权限与团队协作
5. 全量日志中心
6. 技术 JSON 配置面板
7. SaaS 式多租户资产页

这些都可以留给后端能力，前端不需要表现出来。

---

## 10. 设计师交付目标

设计师最终应画出的是：

1. 一个非常干净的企业集成页
2. 一个当前接入总览卡
3. 一个接入方式列表
4. 一个右侧配置抽屉
5. 每种接入方式的结构化配置表单
6. 按接入方式变化的调试区

不是画：

1. 平台管理后台
2. 多实例运维台
3. 复杂机器人运营中心

---

## 11. 一句话交付说明

> 请将企业集成设计为“本地桌面应用中的个人接入工具”：主页面只展示当前平台下的当前接入状态和可用接入方式；配置在右侧抽屉完成；所有配置采用结构化表单；机密字段不明文展示；页面强调轻量、个人化、可快速完成配置与测试，而不是 SaaS 式平台管理后台。
