## 一期目标

飞书第一期要完成的是一组完整能力，而不是只打最小链路：

- 飞书中的消息入口能够调用我们自己的知识库能力
- 系统可以把结果主动通知回飞书
- 飞书文档可以进入知识库导入链路
- 飞书组织信息可以进入本地组织同步链路
- 飞书身份可以进入本地身份绑定 / 登录主链
- 飞书消息资产可以进入归档与全文索引链路

一句话范围：

```text
飞书消息 -> 本地知识库调用 -> 飞书回复
知识库事件 -> 飞书通知
飞书文档 -> 知识库导入
飞书组织/身份 -> 本地账户与权限
飞书消息资产 -> 归档与检索
```

前提约束：

- 当前项目默认运行形态是本地桌面应用
- 当前核心限制是没有公网域名
- 因此一期只采用“不依赖公网回调”的飞书接入方式

## 一期范围

第一期纳入以下 capability 与业务能力：

- `lark.bot`
- `lark.webhook`
- `lark.knowledge_source`
- `lark.identity`
- `lark.org_sync`
- `message_archive`
- `message_search`

其中：

- `lark.bot`
  - 问答入口
- `lark.webhook`
  - 通知出口
- `lark.knowledge_source`
  - 飞书文档接入知识库
- `lark.identity`
  - 身份绑定 / 登录主链
- `lark.org_sync`
  - 组织架构同步
- `message_archive`
  - 飞书消息归档
- `message_search`
  - 归档消息全文索引与检索

#### 飞书后台拿参清单

##### 1. `lark.bot` 长连接问答入口

你要去飞书开放平台后台拿这些参数：

- `App ID`
- `App Secret`

后台路径建议按下面走：

1. 进入飞书开放平台，创建或打开一个自建应用。
2. 打开该应用的 `凭证与基础信息` 页面。
3. 记录：
   - `App ID`
   - `App Secret`
4. 打开应用能力配置，确认已开启 `机器人` 能力。
5. 打开 `事件订阅`。
6. 接收方式选择 `长连接`，不要选 `请求网址 / Webhook`。
7. 在事件列表里勾选后续问答要用到的消息事件。
8. 把应用发布或安装到目标企业 / 测试范围。
9. 把机器人拉进测试群或确认可在目标会话中使用。

说明：

- `tenant_access_token` 不需要你手工去后台复制，它由程序使用 `App ID + App Secret` 动态换取。
- 一期在“无公网域名”前提下，不需要配置回调 URL。

##### 2. `lark.webhook` 通知出口

你要去目标飞书群里拿这些参数：

- `Webhook URL`
- 如启用了安全设置，再额外记录：
  - `签名密钥` 或对应安全配置
  - `关键词` 规则（如果群机器人启用了关键词校验）

后台路径建议按下面走：

1. 打开目标群聊。
2. 进入群机器人的添加 / 管理入口。
3. 添加 `自定义机器人`。
4. 创建完成后复制机器人提供的 `Webhook URL`。
5. 如果你启用了安全设置，再把对应的密钥或规则一并记下。

说明：

- `lark.webhook` 只负责把消息推送到飞书群，不负责接收用户提问。
- 这条链路不需要公网域名。

##### 3. 系统侧你还要准备

- 一个要绑定的知识库 `knowledgeBaseId`
- 一组用于本地联调的测试会话
- 本地服务出网能力
- 身份绑定后的本地用户映射表设计
- 组织同步后的本地部门 / 成员落库方案
- 文档导入后的知识库存储与去重策略
- 消息归档后的索引与查询存储方案

## 一期不做

第一期明确不做下面这些：

- [ ] 基于公网回调 URL 的 webhook 事件订阅接入

- [ ] 多租户运营台
- [ ] 飞书端嵌入式完整前端
- [ ] 跨平台超级统一抽象

## 验收标准

第一期完成的判定标准应非常具体：

### 1. 问答入口验收

- [ ] 用户能在飞书里向机器人发一条消息
- [ ] 本地服务能收到该消息
- [ ] 本地知识库链路被调用
- [ ] 飞书里能收到机器人回复

### 2. 通知出口验收

- [ ] 能从设置页发送飞书测试通知
- [ ] 能从业务事件触发飞书通知

### 3. 知识源验收

- [ ] 能从飞书选择文档导入知识库
- [ ] 导入后能进入现有知识库链路

### 4. 身份与组织验收

- [ ] 能完成飞书身份绑定
- [ ] 能完成飞书组织同步
- [ ] 同步结果能被本地权限或知识库可见性消费

### 5. 归档与检索验收

- [ ] 能完成飞书消息归档
- [ ] 能按关键词检索已归档消息

### 6. 边界验收

- [ ] 飞书没有变成独立问答系统
- [ ] 知识库仍然以本地系统为真相
- [ ] 飞书没有替代本地 Chat 主界面
- [ ] 整条一期链路不依赖公网域名
- [ ] 一期全部既定能力已经纳入统一集成模型

## 完成定义

如果只用一句话定义“一期做完”，那就是：

> 飞书机器人已经可以稳定调用我们的知识库回答问题，系统也可以把知识库相关结果主动通知回飞书；同时飞书文档导入、身份绑定、组织同步、消息归档和全文索引也都已经进入统一的一期交付范围并具备可验收结果。

## 建议实施顺序

1. 先做 `lark.bot` 和 `lark.webhook`，打通问答入口与通知出口。
2. 再做 `lark.knowledge_source`、`lark.identity`、`lark.org_sync`。
3. 最后补消息归档、全文索引、设置页、状态展示和验收文档。

## 无公网域名约束下的结论

- `lark.bot` 只走长连接模式
- `lark.webhook` 可直接作为通知出口
- 一期不接 webhook 回调型事件订阅
- 身份绑定 / 登录主链若涉及依赖回调域名的官方路径，需要在无公网域名前提下定义替代接法或桥接方案
