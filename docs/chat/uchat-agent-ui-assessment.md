## 单点真相范围

这页只回答一个问题：

- 以当前 `uChat` 前端交互来看，它距离“市面上成熟智能体界面”还有多远

这不是后端设计文档，也不是实现清单。它是当前产品形态的界面评估页。

## 评估前提

这里的“智能体界面”不指单纯有一个聊天框，而指用户在聊天过程中能够稳定感知这些事情：

- 当前有哪些能力可用
- 系统正在做什么
- 为什么这样做
- 失败时卡在了哪里
- 当前线程、工作空间、角色、知识库、工具之间是什么关系

如果这些感知不稳定，就不能算成熟智能体界面。

## 当前结论

一句话结论：

`uChat` 现在已经不只是普通聊天界面，而是具备了智能体界面的骨架；但还不能说已经达到市面上成熟智能体产品的交互完成度。

更准确地说：

- 普通聊天体验：已具备
- `Role + RAG + Tool trace` 复合交互：已具备第一版
- 成熟智能体界面的统一感、可解释性、长期状态感：仍未收口

## 已具备

### 1. 普通聊天主界面已稳定成立

当前 `uChat` 已具备稳定的聊天主界面结构：

- 线程列表
- 当前线程消息区
- 输入区
- 线程切换
- 基本运行态反馈

这意味着它已经脱离“功能验证页”阶段，进入可持续演进的产品界面阶段。

### 2. 角色、知识库、工具执行痕迹已经进入聊天界面

当前聊天界面不再只是纯文本对话：

- `Role` 已接到线程级交互
- `RAG` 已接到聊天链路
- tool 调用过程已能进入 execution trace / timeline

这点很关键，因为它说明 `uChat` 已经具备多上下文协同的界面雏形。

### 3. 侧边栏开始具备可插拔能力

`uChat` 侧边栏入口已开始按可插拔方式组织，而不是继续堆硬编码按钮。

这意味着后续新增：

- 聊天搜索
- 工作空间
- 未来 Agent 相关入口

都有继续沿统一入口 contract 演进的基础。

### 4. 工作空间语义已经进入聊天组织层

工作空间虽然还不是完整 Agent workspace，但它已经把聊天组织方式从“单纯线程列表”推进到“分组上下文”。

这是走向智能体产品界面的必要一步。

## 接近具备

### 1. 执行过程可见性

当前已经有 execution trace / timeline，这比普通聊天产品强很多。

但它还只是“有过程展示”，还不是“过程感知完整”。

还差的主要是：

- 用户是否能一眼看懂当前阶段
- 节点名称是否足够统一
- 不同能力是否都走同一展示语言
- 失败态是否足够清楚

所以它处于“方向正确，但产品表达还没完全成熟”的阶段。

### 2. 工作空间交互骨架

当前工作空间已经具备：

- 分组展示
- 展开收起
- 创建删除
- 激活线程归属

但它当前仍主要是聊天列表组织层，不是完整的项目工作区。

距离成熟智能体 workspace 还差：

- 文件上下文
- 任务上下文
- 工具上下文
- 多线程共享状态

### 3. 角色感知

当前 Role 已经能改变：

- 聊天头像
- 回复中状态
- 请求侧上下文

这说明角色不再只是设置页数据。

但从用户认知上看，Role 还没有完全形成“长期人格状态面板”或“线程上下文身份卡”的产品感。

## 明显缺失

### 1. 能力可用性表达还不够稳定

成熟智能体界面通常要让用户明确知道：

- 当前能不能搜索
- 当前能不能调用本地工具
- 当前为什么不能调用某个工具
- 当前线程启用了哪些增强能力

现在这部分认知仍偏分散，更多依赖用户自己猜，或者依赖实现细节。

### 2. 工具决策反馈还不够强

目前已有工具调用链路，但“为什么调用这个工具、调用完成后何时该收口回答”这层用户感知还不够稳定。

也就是说，现在更像“能看到工具跑过”，还不是“能理解 agent 如何决策”。

### 3. 长期状态还没有统一产品表达

当前已经逐步接入或预留了这些长期状态能力：

- Role
- Summary
- 未来 Memory
- Workspace

但用户界面上还没有把这些整合成一个统一的“线程长期上下文”认知。

这会带来两个问题：

- 用户不知道哪些状态正在持续影响当前回复
- 用户很难判断某次回复到底受了哪层上下文影响

### 4. 失败态与恢复路径还不够产品化

成熟智能体界面通常需要更明确的失败反馈：

- 是模型失败
- 是工具失败
- 是知识库没命中
- 是权限不足
- 是工作空间上下文无效

并且最好能提供：

- 重试
- 忽略后继续
- 切回纯聊天

当前 `uChat` 这部分还在工程可用阶段，没有完全进入产品化状态。

## 为什么现在还不能说“已经具备市面成熟智能体界面”

不是因为功能少，而是因为“统一感”还不够。

成熟智能体界面的核心，不是功能堆得多，而是这些功能在用户看来属于同一套系统：

- 同一套能力表达
- 同一套执行过程表达
- 同一套上下文表达
- 同一套失败恢复表达

`uChat` 现在最大的优点是：

- 骨架已经出来了
- 主方向基本正确

`uChat` 现在最大的缺口是：

- 这些能力还没有完全收口成同一套产品语言

## 当前判断

如果用产品成熟度来粗分：

- 不是“普通聊天 UI”
- 也还不是“成熟 Agent 产品 UI”
- 更像“智能体聊天底座 + 第一版产品交互层”

这个定位更接近当前真实状态。

## 建议优先级

如果继续从前端交互角度收口，优先级建议如下：

1. 统一能力可用性表达
2. 统一 execution trace 的节点语言与失败态
3. 把 `Role / Summary / Memory / Workspace` 收口为“线程长期上下文”的同一产品认知
4. 再考虑更完整的 Agent workspace 形态

## 对当前开发判断的影响

这份评估意味着：

- 当前不该把 `uChat` 当成普通聊天页来修修补补
- 也不该误判为“已经完整具备成熟智能体交互”

更合适的策略是：

- 保留当前骨架
- 继续沿统一 contract 收口
- 尽量避免为每个新能力再造一套私有 UI
